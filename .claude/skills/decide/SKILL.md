# decide — プライベート意思決定サポート

ASD特性を持つユーザーのプライベート領域（お金の使い道・人間関係・生活の選択）における意思決定を、構造化された対話で支援し、判断結果を `decisions/` に markdown ファイルとして保存するスキル。

## 設計原則

- 曖昧さを排除し、明確な基準とルールで判断を導く
- 感情ベースではなく事実ベースの質問をする
- スコアリングで白黒つける。「考えてみて」で終わらせない
- **急がない。情報が揃うまでスコアリングに入らない**

## 対話フロー

### Phase 1: 状況の深掘り

まずユーザーの話をじっくり聞く。AskUserQuestion の選択肢形式は使わず、**自由記述で答えてもらう**形式で以下を掘り下げる。1つの質問につき1メッセージ。

1. **何についての判断か？** — 引数で渡されていればスキップ
2. **カテゴリ分類** — AskUserQuestion で確認:
   - 「お金を使う判断」（購入・入会・投資など）
   - 「人間関係の判断」（誘いへの返答・関係の距離感など）
   - 「生活の変化」（引っ越し・習慣の変更など）
3. **比較型かどうか？** — 「AとBどちらにするか」型の判断か、「やるかやらないか」型の判断かを判別する。比較型の場合、選択肢を明確にする。
4. **状況の掘り下げ（最低3往復）** — 以下のような質問を自由記述で聞く:
   - 「体験してみてどうだった？何が印象に残っている？」
   - 「迷っているポイントを全部挙げてみて」
   - 「理想的にはどうなったら一番嬉しい？」
   - 「何が不安？最悪のシナリオは？」
   - 比較型の場合: 「それぞれの良かった点・気になった点は？」
   - ユーザーの回答に応じて追加で掘り下げる

**Phase 1 の完了条件**: 以下の3つが揃ったとき:
- 判断の背景と文脈が十分に理解できた
- ユーザー自身の主観的な感想・懸念が出揃った
- 比較型の場合、各選択肢の主観的な評価が出た

### Phase 2: 客観情報の収集

WebSearch / WebFetch を使って、判断に必要な客観的事実を自動で集める。ユーザーに調べさせない。

#### 収集する情報の例

- **入会・購入系**: 料金（月額・年額・入会金）、解約条件、営業時間、立地・アクセス、口コミ・評判
- **サービス比較**: 各プランの違い、公式サイトの情報、第三者レビュー
- **生活の変化**: 相場感、一般的なメリット・デメリット、経験者の声

#### 手順

1. 比較対象ごとに WebSearch で公式サイト・レビュー等を検索する
2. WebFetch で公式サイトから料金・条件などの具体的な数値を取得する
3. 集めた情報を **客観情報比較表** としてユーザーに提示する
4. 「この情報で合ってる？抜けている情報はある？」と確認する

**重要**: Web検索で十分な情報が得られない場合は、ユーザーに「〇〇の情報がWebで見つからなかった。知っていたら教えて」と聞く。推測で埋めない。

### Phase 3: 即決チェック（お金カテゴリの場合）

金額を確認する（Phase 2 で判明していればそれを使う）。

- **1万円以下** → 即決ルール発動:
  - 「実験費。やってみろ。失敗しても1万円以下の授業料。」と伝える
  - 結果ファイルには `result: go（即決ルール）` で記録
  - Phase 6 へスキップ

- **1万円超** → Phase 4 へ

### Phase 4: 構造化スコアリング

客観情報と主観情報が揃った段階で初めてスコアリングに入る。各項目を AskUserQuestion で1つずつ聞く。Phase 2 で得た客観情報がある場合は、選択肢の説明にその情報を反映させる。

#### 「やるかやらないか」型: お金を使う判断 / 生活の変化

以下5項目を各1〜3点で評価する（合計5〜15点）:

| # | 項目 | 1点 | 2点 | 3点 |
|---|------|-----|-----|-----|
| 1 | **可逆性**: やめられるか？ | 不可逆（手術・タトゥーなど） | 条件付きで戻せる（違約金あり等） | いつでもやめられる（月額解約等） |
| 2 | **最悪ケース耐性**: 完全に失敗したら？ | 生活に支障が出る | 痛いが耐えられる | 笑い話にできる |
| 3 | **不作為の後悔**: 1年後「やらなかった」を後悔するか？ | しない | 少しする | 強く後悔する |
| 4 | **代替手段**: もっと小さく試せるか？ | 小さく試せる（まずそっちをやれ） | 部分的に可能 | これが最小単位 |
| 5 | **現在の余裕**: 金銭・時間・精神に余裕があるか？ | 余裕なし | どれか1つ不足 | 全部ある |

**判定基準**:
- **12〜15点**: → **やる**。条件は揃っている。
- **9〜11点**: → **小さく試す**。代替手段やお試し期間を探す。
- **5〜8点**: → **今はやらない**。条件が整ったら再検討。

#### 「AとBどちらか」型: 比較スコアリング

Phase 1〜2 で出た判断軸を使い、比較表を作る。

1. Phase 1 の深掘りで出た **ユーザーにとって重要な判断軸** を3〜5個抽出する
2. 各軸について「どちらが優れているか」を AskUserQuestion で確認する（客観情報で明白なものは自動記入）
3. AskUserQuestion で「この中で最も重要な軸はどれ？」を聞く（重み付け）
4. 勝ち数 + 重要軸の勝敗で判定する

#### 人間関係の判断

以下4項目を AskUserQuestion で確認する:

| # | 項目 | 質問 | 選択肢 |
|---|------|------|--------|
| 1 | **関係への影響** | 断ったら関係は壊れるか？ | 壊れる / 多少ギクシャク / 影響なし |
| 2 | **精神的コスト** | 受けた場合の精神的負担は？ | 大きい / まあまあ / 小さい |
| 3 | **本音** | 本音はどっち？ | 行きたい・やりたい / 行きたくない・やりたくない / わからない |
| 4 | **代替案** | 別の形で応じられるか？ | ある（具体的に） / ない |

**判定ロジック**:
- 本音が「行きたい」＋ 精神的コスト「小さい」 → **やる**
- 本音が「行きたくない」＋ 関係「影響なし」 → **断る**（断り方を提案する）
- 本音が「行きたくない」＋ 関係「壊れる/ギクシャク」 → **代替案を提案**
- 本音が「わからない」 → 精神的コストと関係への影響で判定
- 代替案がある場合 → 代替案を最優先で提案

### Phase 5: 判定の提示

スコアリング結果をまとめて提示する。

- 比較型の場合: 比較表 + 判定理由を明示する
- 結果ファイルに含める情報（客観情報・主観情報・スコア）を一覧で見せる
- 「この判定で記録していいか？」と確認する

### Phase 6: 結果ファイルの生成

`decisions/` ディレクトリに以下のフォーマットで markdown ファイルを保存する。
ファイル名: `YYYY-MM-DD-テーマの短い要約.md`

```markdown
---
tags:
  - decision
created: YYYY-MM-DD
status: active
category: money | relationship | life-change
result: go | try-small | not-now | decline
---

# [判断テーマ]

## 状況

[ユーザーが述べた状況の要約]

## 客観情報

[Phase 2 で収集した事実ベースの情報。比較型の場合は比較表]

## スコアリング

[「やるかやらないか」型の場合]
| 項目 | スコア | メモ |
|------|--------|------|
| ... | .../3 | ... |

**合計: XX/15**

[比較型の場合]
| 判断軸 | 選択肢A | 選択肢B | 重要度 |
|--------|---------|---------|--------|
| ... | ... | ... | ★/☆ |

## 判定: [go / try-small / not-now / 選択肢名]

[判定理由の要約。2〜3文で。]

## 次のアクション

- [ ] [具体的な次の一手]
```

### 補足ルール

- 情報収集の時間制限を伝える:
  - 1〜5万円: 「調べるのは最大3日。それ以降は今ある情報で決めろ。」
  - 5〜10万円: 「最大1週間。」
  - 10万円超: 「最大2週間。」
- 「もっと調べたい」と言われたら: 「何を調べれば判断が変わるか？」を聞く。答えられなければ情報は十分。
- 判定が出た後に迷っていたら: 「判定はルールに基づいている。感情で覆すなら、それは感情の問題であって情報の問題ではない。」と伝える。
