## コンポーネントの定義
- 関数コンポーネント
JavaScriptの関数として定義する。関数コンポーネントはpropsを受け取り、JSXを返す
e.g.)
//関数の定義は"大文字"ではじめよ

//関数宣言の時

	return <h1>Hello</h1>; //JSXを返す
}

//アロー関数の時
const Example = () => {
	return <h1>Hello Components</h1>;
}
	
	
- コンポーネントの実行（呼び出し)
<Welcome />


//複数行に渡る場合には、returnキーワードの性質上、改行すると undefinedが返ってくる。なので、returnの後には改行せずにreturnする値を指定すること。

# Fragment
JSXで返す要素は、1つのルート要素にまとまっている必要がある。
つまり、
<div className="component">
	<h3>Hello Component</h3>
</div>
<h3>hello world</h3>
<p>Lorem ipsum dolor sit amet consectetur adipisicing elit. Aut exercitationem qui suscipit et minima neque magni nam facere consequatur modi ut accusantium quo alias eaque, rerum sint numquam quia placeat.</p>

以上のような場合には、div要素とh3要素とp要素が階層の最上にあるが、JSXではこのようなルート(最上位になる要素)が複数あるものを許容しないので、何もないdivだとでwrapしてあげる必要があります。

## Fragment

# useState(React Hooksの一つ)
Reactの関数の一つ。ユーザの入力値に応じて、リアルタイムで出力の値を再生成することができる。

- このメソッドの返り値は、2つ要素を持つ配列となっています。
その中身について
	- [0]には、参照用の値が格納されています。(useState()呼び出し時の引数に初期値を取ると、初期値が最初格納されます。)。//Reactの内部に状態を保持してもらうお願いしている。
	- [1]には、値の更新関数が格納されています。この関数に更新したい値を渡す形で,[0]の値を更新することができる。
//より内部的な話をすると、React側で、useState[0]の状態を保持してくれていて、useState[1]を介して、更新もできる。また更新した際には、自身のコンポーネントの中身を再実行(render)もしてくれる。
// このようにReact内部に保持されるコンポーネントに紐づいた値をstateと呼ぶ。

e.g.)//分割代入も駆使してuseStateを使う例
```
//Reactライブラリから取ってきて、
import { useState } from "react";


const Example = () => {
  let [value, setValue] = useState();
  return (
    <>
      <input 
      type="text"
      onChange={(e) => {
        setValue(e.target.value) //setVauleはuseState[1]の部分。引数にとった値に、[0]の値を更新する。
      }} /> = {value} //valueはuseState[0]の部分、参照用の値
    </>
  );
};


```


# State(ステートについて)
もし、画面が再renderされるような場合にも、何かしらの値を保持していたい時に、保持できる性質を広義に"状態を持つ(stateful)"を言う。
- Reactにおいて、stateはコンポーネント単位に保持、更新される。

まず、うまく行かなかった方法を見て、そこから学んでいこうと思う。
e.g.)
```
let displayValue;
onChange={(e) => {
        displayValue = e.target.value;
}} /> = {displayValue}
```

//これは、イベントリスナーによって、変数displayValueの値が更新されている。がしかし、実際にコードが実行された部分は、イベントハンドラのコールバック関数部分である。
つまり、displayValueの参照先は異なる訳ではなく、イベントリスナーの外部分は、イベント発火によって、再render(実行)されてないので、古い参照結果を表示させ続けることになっている。
//これを再度実行しようとするたびに、e.target.valueの値は保持されない。つまり、状態を持たないので、永遠に変更を反映できず、初期化され続ける。

# Stateとrendering(ステートとレンダリング)
前述もしてあるけれども、Stateを使うことによって、コンポーネント自体のサイレンダーしてくれる。not only the part of EventLister

# Stateを使う際に注意すべき点
1. 呼び出す位置について
useStateとは基本的に、**コンポーネントのトップレベル**もしくは、カスタムフック内でのみ呼び出すことができます。(即ち{}の部分 ifやfor文中には書けない。)
//<- これは、Reactの各コンポーネントにおけるstateの保持の仕方によるもの。
あるコンポーネントで、3つstateを作った時、それら3つは1,2,3番目と順番を持って(List?)管理される。if文などの条件分岐によって1個目がない？！みたいなことになると、エラーが発生するから。

2. 即時更新ではない!?
更新関数で、React内部に保持されている値を更新します。この時、更新関数からstateの保持している値を更新するように依頼はしているが、即時実行ではない。
/*ただ、これは非同期に処理される */ cf) 05_state_and_event/050_prev_state

```
const Example = () => {
  const [count ,setCount] = useState(0);
  const countUp = () => {
    setCount(count + 1);
    console.log(count); //countの値は初期値のままで、再レンダーまで更新されない。
  };
  const countDown = () => {
    setCount(count - 1);
  };
  return (
    <>
      <p>現在のカウント数: {count}</p>
      <button onClick={ countUp }>+</button>
      <button onClick={ countDown }>-</button>
    </>
  )
};

```

useState[1]の更新関数は、非同期に処理されるが、再renderされる前の現在の状態におけるstateを更新したい場合には、引数にコールバック関数を取ることができる。

```
const countUp = () => {
    setCount(count + 1); --①
	setCount(prev => prev + 1); --②
};

```
//コールバック関数の引数に渡ってくる部分(今だとprev)は,直前に更新依頼されたstateの値を参照できる。
即ち、上の例だと、実際にイベントが発火した際には、

①は state = 1;
②は, state = func(state); 

という時系列で処理が行われる。

//なぜか？
- そもそも、Reactは、Fiberツリー(h1などのHTMl要素とReact要素からなるツリー)の差分検出によって、本物のDOMを書き換えている。
- 書き換えの際に、親nodeから見た末尾の要素が追加された場合には、子要素をそれぞれ比較し、新たに追加された要素を実際のDOMに反映する。
- しかし、親nodeから見た先頭の要素が追加された場合を考えると、実際には先頭に追加されただけでも、Reactは/*子要素をツリー構造の位置をもとにした子要素の中身*/で比較し、今までのツリー上の位置のものが全て変わったと認識する。それによって、すべての子要素を洗いがえることになる。(配列として比較されるイメージ)

- そこで、keyプロパティをつける。
この時Reactの処理は、子要素の位置と中身での比較ではなく、keyによる変化前後の紐付けをしているため、新たなkeyの値を持つ要素が追加された、もしくは削除された？かをもとに更新する。
(つまり、同じkeyを持つ要素で中身が同じならば更新しない。)
(連想配列を作るイメージ、つまり、keyによって要素の増減を見て？同一keyのものの中で変化があるかもみる。なければそのまま)
それによって、実際の差分だけを更新することができるようになります。

- 配列のインデックスをなるべく使わないこと。

## マルチセレクトボックス
-  HTMLの場合には、checked属性自体が付与されているがどうか。によって判断される。しかしJSXにおいては、それぞれのエレメントのcheckedプロパティの真偽値によって判定される。
import { useState } from "react";

const Example = () => {
  const [fruits, setFruits] = useState([
    { label: "Apple", value: 100, checked: false },
    { label: "Banana", value: 200, checked: false },
    { label: "Cherry", value: 300, checked: false },
  ]);

  const [sum, setSum] = useState(0);

  const handleChange = (e) => {
    const newFruits = fruits.map(fruit => {
      const newFruit = { ...fruit};
      if( newFruit.label === e.target.value) {
        newFruit.checked = !fruit.checked;
      }
      return newFruit;
    })

      setFruits(newFruits);

      //俺の回答&&を使ったショートサーキット
      let sumVal = 0;
      newFruits.forEach(fruit => {
        fruit.checked && (sumVal += fruit.value);
      });
      //Plainな記法
      // newFruits.forEach(fruit => {
      //   if(fruit.checked) {
      //     sumVal += fruit.value;
      //   }
      // });
      //filterで配列を置換してから、forEachでvalueの値を回す。
      // newFruits
      //   .filter(fruit => fruit.checked)
      //   .forEach(fruit => (sumVal += fruit.value));
      setSum(sumVal);
    };

  return (
    <div>
      {fruits.map(fruit => {
        return (
            <div key={fruit.label}>
              <input 
                id={fruit.label}
                type="checkbox"
                value={fruit.label}
                checked={fruit.checked}
                onChange={handleChange} 
              />
              <label htmlFor={fruit.label}>
                {fruit.label}:{fruit.value}
              </label>
            </div>
        );
      })}
      <div>合計：{sum}</div>
    </div>
  );
};

export default Example;

# 関数型プログラミングの理解(Functional Programming)

## 手続き型(命令)プログラミング
ブラウザなどへ命令を手順通りに記述していく手法。

- 関数型プログラミングとは、なるべく手続き型の制御を関数に分離(隠蔽)し、やりたいことに集中できるようにするプログラミング手法。

e.g.)
let nums = [1,2,3];
let doubleNums = nums.map(num => num * 2);
ループ制御:->mapメソッドに担当させる。
やりたいこと: ->関数で定義。(開発者が担当)

# 関数型プログラミングのメリット
1. コードの可読性の向上
2. 拡張性/再利用性の向上
3. テスト性の向上
4. モジュール化の向上
5. Tree Shakingの向上(本番用に無駄なコードを削除する機能)

# 関数型プログラミングにおけるキーワード

- *不変性(immutability)

import { useReducer } from "react";

const CALC_OPTIONS = ["add", "minus", "divide", "multiply"];

const reducer = (state, {type, payload}) => {
  switch (type) {
    case "change":
      const {name, value} = payload;
      return { ...state, [name]: value};
    case "add":
      return { ...state, result: state.a + state.b }
    case "minus":
      return { ...state, result: state.a - state.b }
    case "multiply":
      return { ...state, result: state.a * state.b }
    case "divide":
      return { ...state, result: state.a / state.b }
    default:
      throw new Error("Unknown type");
  };
};

const Example = () => {
  const initState = {
    a: 1,
    b: 2,
    result: 3,
  };

  const [state, dispatch] = useReducer(reducer, initState);

  const calculate = (e) => {
    dispatch({type: e.target.value}) //ちなみに、todo: todoみたいな場合には省略できるらしいよ。
  };
//payloadを使うことで、eのイベントの発生した要素の別のプロパティを

    dispatch({type: "change", payload: {name: e.target.name, value: parseInt(e.target.value)}})
  }

  return (
    <>
      <div>
        a:
        <input
          type="number"
          name="a"
          value={state.a}
          onChange={numChangeHandler}
        />
      </div>
      <div>
        b:
        <input
          type="number"
          name="b"
          value={state.b}
          onChange={numChangeHandler}
        />
      </div>
      <select value={state.type} onChange={calculate}>
        {CALC_OPTIONS.map(calc => {
          return (
          <option key={calc} value={calc}>
            {calc}
          </option>
          )
        })};
      </select>
      <h1>結果：{state.result}</h1>
    </>
  );
};

# useContextを使用する上での注意点
- useContextを使用してあるstateを更新した場合には、当該のstateを使用している全てのコンポーネントにおいて再レンダーされる。ということ。

## useEffectの第二引数について
useEffectの第二引数に当たる配列は、依存配列と呼ばれる。この依存配列には、コールバック関数の実行タイミングを制御する依存データが入る。
例えば、なにかしらのStateをこの配列に入れた時、これは、Stateの更新のタイミングに応じて、コールバック関数を再度呼び出す。
つまり、再実行させる必要がない時には、[]再実行の制御は不要で、逆に1秒ごとに更新させたいのであれば、一秒ごとに更新する値を用意するが吉。若しくは[]ではなく、一度だけ呼び出す関数側で制御するのもアリ。

### 注意点
useEffectのコールバック関数内で、依存配列と依存性のある関数を呼んだ場合には[]、再帰呼び出しが発生しますので、やめてください。

## useEffectの第一引数にわたるコールバック関数の戻り値。//依存配列が空の場合
この戻り値は、useEffectが呼び出されたコンポーネントが消滅した際に、一度だけ呼び出される関数に当たる。
これによって、指定されたコンポーネントが消滅した際の挙動も制御できる。
e.g.)
  useEffect(() => {
    console.log('init');
    window.setInterval(() => {
      setTime(prev => prev + 1);
    }, 1000);
    return () => {
      console.log("done");
    }
}, [])

## useEffectの第一引数にわたるコールバック関数の戻り値。//依存配列に定期更新される何かが入っている場合
この時コールバック関数の挙動としては、まず一度、第一引数のコールバック関数が呼ばれる。その後、依存配列の制御のともなって、再実行される時に、return以後の戻りに指定された処理も実行されるので、
callback --> return/callback => return/callbackの順で、依存配列の制御に倣ってよびださられる。

e.g.)
  useEffect(() => {
    console.log('updated')
    document.title = 'counter:' + time;
    window.localStorage.setItem('time-key-end', time)

    return () => {
      
      console.log("update end")
    }
  }, [time]);




## useEffectのクリーンナップ処理
依存配列が設定されていない場合に、一度useEffectに呼び出された、コールバック関数は再度実行されることはない。
しかし、clearIntervalなどで定時実行させるものを停止したい場合には、return 以降に、clearIntervalを仕込んで、コンポーネントの消滅にトリガーしてreturnが実行。その中に記述されたclearIntervalによって、クリーンナップできるという設定に使える。


## useEffectの実行されるタイミングについて
1-[依存配列が空の時]
e.g.)
useEffect(() => {
 //callback処理
	return () => {
	//cleanup処理
  }
}, []);

1. コンポーネントがMountedされた時に、callback処理が呼び出されて、
2. UnmountedされたタイミングでcleanUp処理が実行される。


2-[依存配列に値がある時]
useEffect(() => {
	//callback処理
	return () => {
	//cleanup処理
  }
}, [/*依存値*/]);

1. コンポーネントがMountedされた時に、callback処理が呼び出されて、
2. stateの更新などによって、コンポーネントがUpdated(再レンダー&DOM更新)され、依存値に指定していた値が更新されていれば、
	a. cleanup処理が実行され
	b. callback処理が実行される
3. 2の操作がUpdatedの都度実行されて、
4. UnmountedされたタイミングでcleanUp処理が実行される。

3-[依存配列が省略されている時]
2-のケースとの違いは、stateの更新などによるコンポーネントのupdated(再レンダー)が起きた際に、依存値がないため、依存値の更新にかかわらず。(つまり、Updatedされるたびに)cleanup処理とcallback処理が呼び出される。



## useLayoutEffectについて
- useLayoutEffectは、useEffectよりも先に実行される。
よって、
const _time = parseInt(window.localStorage.getItem('time-key'));
のような処理を記述することで、ブラウザのストレージに対して、保持されていた値を、useEffectが再度マウントされる際（今回はExample >TimerコンポーネントのExampleコンポに含まれるtimeというstateによって、毎秒再レンダリングされている。)、useEffectに先んじて、ストレージにsetされていた値をとりに行くことで、状態の保持を有効にしている。

- 画面上での表示stateなどによって、更新がかかった際には、実際のレンダーよりも先んじて処理を実行する。
例えば以下のような例。

import { useLayoutEffect, useEffect, useState, useRef } from "react";

const Random = () => {
  const [state, setState] = useState(0);

  useLayoutEffect(() => {
    if (state === 0) {
      setState(Math.floor(Math.random() * 300));
    }
  }, [state]);

  return (
    <button
      className="effect-btn"
      onClick={() => setState(0)}
      style={{ fontSize: "2.5em" }}
    >
      state: {state}
    </button>
  );
};
export default Random;

1. 初回のレンダー時には、useLayoutEffectが実行されて、state===0によって、ランダムが数字がセット。
表示される。

2. onClickイベントとしてStateをゼロに変更するようになっているので、この時、まず、stateの値に変更依頼が予約される。
3. 再レンダーが行われる時に、
4.この時通常のuseEffectであれば、依存配列にstateが指定されるため、非同期的にsetState更新関数で、ランダムな値を設定し直し始める。その更新の前後差によって、実際にレンダーされる値が、新たなランダムな値であったり、0が見え隠れしたりする。

//この時にuseLayoutEffectを使用すると、stateの更新による再レンダーに先んじて、内部のコールバック関数を実行する。

//Timerの機能を保つコンポーネントの一部分を切り出してuseTimerという関数に切り出す。
import { useState, useEffect, useLayoutEffect } from 'react';
const useTimer = () => {
  const [time, setTime] = useState(0);
  const [isRunning, setIsRunning] = useState(false);

  useEffect(() => {
    // console.log('init');
    let intervalId = null;

    if(isRunning) {
      // console.log('timer start');

      intervalId = window.setInterval(() => {
        // console.log('interval running');
        setTime(prev => prev + 1);
      }, 1000);
    }
    
    return () => {
      window.clearInterval(intervalId)
      // console.log('end');
    }
  }, [isRunning])
  
  useEffect(() => {
    // // console.log('updated');
    
    document.title = 'counter:' + time;
    window.localStorage.setItem('time-key', time);

    return () => {
      // debugger
      // // console.log('updated end');
    }
  }, [time]);

  useLayoutEffect(() => {
    const _time = parseInt(window.localStorage.getItem('time-key'));
    if(!isNaN(_time)) {
      setTime(_time);
    }
  }, [])

  const toggle = () => {
    setIsRunning(prev => !prev);
  }

  const reset = () => {
    setTime(0);
    setIsRunning(false);
  }
//帰り値に、定義済みで、他のコンポーネントで受け取りたい値について設定しておき。
  return {
    toggle,
    reset,
    isRunning,
    time,
  }
};

export default useTimer;

//呼び出し側のファイルにて

    toggle,
    reset,
    isRunning,
    time,
} = useTimer();

## Immerの仕組み。

- mutableな値の例は、Objectや配列などのこと。

### Immerの仕組み
- produce関数()第一引数に、元となるオブジェクトをとり、第二引数にコールバック関数を設定する。

produce(obj, draft => {
  console.log(draft);
});

素晴らしい。元のオブジェクトに変更を破壊的に行わない。
上の例に挙げられているようなhobbies配列の中身、つまりネストされたmutableなオブジェクトについても、immutabilityが保持される。
Immerを使う場合には、関数の機能として、immutabilityを保持した上で返り値も設定してくれるので、ユーザ側でreturnを書くべきではない。

# rendering(レンダリング)の最適化
- トリガー: レンダリングを予約すること
- レンダリング: コンポーネントを実行すること
- コミット: DOMの更新を行うこと


## レンダリングがトリガーされるタイミングについて
1. 初回レンダリング
ルートコンポーネントをHTML(DOM)にマウントした時
<div id="app"></app> <-- root.render(<App />)

2. "state"の値が変更された時
コンポーネントのstateの値が変更された時。
この時に、再レンダリング行われる。再レンダリングが行われる際には、それ以降の子要素すべてに対してもツリー構造上行われる。
//基本的には, stateの前後の値に差が生じた際にレンダリングがスケジュールされる。

### stateの比較(トリガーがされるきっかけ)
更新用関数に渡された値と保持している値を比較し、差異があればレンダリングがスケジュールされる。(値が同一でも、reactの仕様上レンダリングがされることもある。)


### コミットについて
レンダリングのトリガーが行われ、レンダリングをした後には、React要素が更新される。その後、実際のDOMツリーに対しても変更が適応される。


### Object.is(val1, val2)がtrueになる時、それらの変数名は同一のオブジェクトを参照している。つまり同一のオブジェクトを指し示しているため、再レンダーは行われない。

オブジェクトを変数に格納する際には、ミュータブルゆえに一見すると同じ値を代入したとしても、それは異なる場所に、同一の値を持ったオブジェクトが2つ存在し、それぞれに対しての参照として変数名がついているということ。
e.g.)
const A = {val: 1}
const B = {val: 1}
const C = { ...A }
const D = A
Object.is(A, B) //->>false
Object.is(A, C) //->> false
Object.is(A, D) //->> true


数値や文字列などのプリミティブな型のものについては、同一の操作をしたときも、メモリ上では同一の値を自動で参照する？らしい
e.g.)
const A = 1;
const B = 2;
Object.is(A, B) // ->> true

# React.memo
受け取ったpropsの値が同じであれば再レンダリングをスキップする関数。
値の同一性に関してはオブジェクトとして同一かどうか？

e.g.)
import { memo } from "react";
//memo()という関数の第一引数に、コンポーネントを指定すると、propsで受け取るstateやuseContext経由で受け取ったstateが更新された時にのみ、再レンダリングされるようになる。
//親の影響を受けたくない子コンポーネントに対して使う。　

const Child = memo(({ countB }) => {
  console.log("%cChild render", "color: red");
  return (
    <div className="child">
      <h3>子コンポーネント領域</h3>
      <p>ボタンBクリック回数:{countB}</p>
    </div>
  )
});


### memo()の第二引数について
第一引数には、親の影響を受けたくない子コンポーネントを受け取ることで、無駄な再レンダーを防止する。





# useCallback(React Hooks)
関数のメモ化する。
先ほどのmemoでのオブジェクトとしての同一性を用いて、変更があるのかを判定する際に、propsが関数だった際には、再レンダーのたびに同値的なオブジェクト(関数)が再生成されているため、何度も変更があったとして再レンダリングの対象になってしまうう。このことを防ぐための関数。

子コンポーネントに、渡される親コンポーネント内に定義された関数を第一引数に取る形で利用する。

useCallbackの第一引数には、コールバック関数を設定し、第二引数に依存配列をとる。
e.g.)
  const clickHandler = useCallback(() => {
      setCountB((pre) => pre + 1);
    }, [])
//第一引数に渡される関数は、React内部に保持される値となる。再レンダリングの際には、保持された値を返すという制御になる。これによって、子コンポーネント側でmemo()とすると、同一オブジェクトが渡ってくるように設定できる。

## useCallbackの依存配列の使い方
第一引数のコールバック関数内にstateを用いる際には、そのstateの値を依存配列[]に入れると良い。これによって、stateの更新と同時に、React側で保持される関数も更新されるため。

e.g.)//この例では、countBが更新される度、更新関数の値もstateも依存関係を持って更新したい。そうでなければ呼び出されるたびにcountB = 1 + 1 ->> 1になるから。
const clickHandler = useCallback(() => {
		setCountB(countB + 1);
}, [countB])


# useMemo(React Hooks)
あらゆる値をメモ化できる。 メモというのはキャッシュかだと思ってくれて大丈夫。
//ただし、useMemo自体の処理にもコストがかかるため、むやみに使用するのは避けるべき
### React.memoとの相違点
- React.memo: コンポーネントのみをメモ化
- useMemo: あらゆる値をメモ化可能.


//基本書式
useMemo(() => {},[])
第一引数にとった値をメモ化し、[]に含まれた値が更新された時に、中身を再度実行することになる。[]が更新されなければキャッシュされたそれを返す。





# Concurrent Mode(React 18の試験的実装された機能)
同時性モード？
処理の優先順位づけを行うことで、より快適にレスポンスシブな画面を作成するための機能。


## useTransition
パフォーマンスの最適化を行った上でも、処理が重い場合に、優先順位づけをする目的で使用すること。
e.g.)
const [isPending, startTransition] = useTransition();
// 第二引数にとったstartTransitionのコールバック関数に、実行の優先順位を下げたい処理を記述する。
  const changeHandler = (e) => {
    startTransition(() => {
      setFilterVal(e.target.value);
    })
  };

- isPendingは、startTransitionの処理が終了前であればtrue、でなければfalseを返す。



## useDeferredValue
こちらも、先ほどのものと同様に遅延処理されることで、更新させるものの優先順位を設定することができる。

こちらの場合には、引数に遅延処理させたいものをそのまま囲む。
e.g.)

  const filterdItems = dummyItems
  .filter((item) => {
    if (filterVal === "") return true;
    return item.includes(filterVal);
  })
  .map((item) => (
    <li key={item}>{item}</li>
  ));

  const deferredItems = useDeferredValue(filterdItems);
  
  return (
    <>
      <input type="text" onChange={changeHandler} />
      {/* {isPending && <div>Loading...</div>} */}
      <ul>
        {deferredItems}
      </ul>
    </>
  );
};





# REST APIを使ったサーバーとの連携　

RESTという思想に基づいてサーバーとデータのやり取りをする。
現在のwebにおいてかなり一般的に普及した方法。

RESTについての特徴をまとめると、
- リソースごとにURLを定義しておく。
- それに対してHTTPメソッド(get,post,delete,putなど)によってリソースに対する処理を定義する。
- JSONを用いてデータのやり取りをする。

## APIサーバーのモックを作成して、APIとのJSONを使ったやりとりを学ぶ。


## Axios
- getリクエストを行う。
e.g.)
//この時にPromiseオブジェクトが返ってくるので、その時のためにthenかawaitで非同期処理に対応する必要がある。
import { useEffect } from "react";
import axios from "axios";

const Example = () => {
  useEffect(()=> {
    axios.get("http://localhost:3003/user").then((res) => {
			console.log(res);
			console.log(res.data);

  })
  return <></>;
};

	export default Example;
	
};

async awaitについても、useEffect内では、直接関数をasyncにすることはできないので、以下のように書く。

e.g.)

const Example = () => {

### axiosをつかって、JSONデータを取得し、それを表示する。
//Reactでは非同期に取得してきたものに関しては、stateで管理する。
e.g.)
import { useEffect, useState } from "react";
import axios from "axios";


const Example = () => {
  //stateの初期値を[]しないと、undefinedとなり、mapメソッド持たないためにエラーが発生する。
  const [ users, setUsers] = useState([]);
  useEffect(() => {
    const getUser = async () => {
      const res = await axios.get("http://localhost:3003/user")
      // console.log(res.data)
      setUsers(res.data);

    }
    getUser();
  }, []);
  
  return (
    <div>
        {users.map(user => {
          return (
            <div key={user.id}>
              <h3>{user.username}</h3>
              <p>age: {user.age}</p>
              <p>hobby: {user.hobbies.join(",")}</p>
            </div>
          )
        })}
    </div>
  )
};

export default Example;

## Dynamicインポート(JSの機能)
importキーワードでimportする時、staticインポートとも呼ばれる。
importの関数を使用し、引数にパスとなる文字列を渡すことで成立する。
//引数の文字列を変更することで、importするものを変更できるので、Dynamic(動的インポートと呼ばれている。)
e.g.)
const Example = () => {
//非同期関数を定義しasync,awaitでPromiseオブジェクトを扱う例。
  const dynamicImport = async () => {
    const module = await import("./add")
    console.log(module);
  };
  dynamicImport();
//thenを使って、importした返り値であるPromiseオブジェクトを扱う例。
  import("./add").then(module => {
    console.log(module);
  })
};

## ReactでDynamicインポートを使用する方法。
インポートしたいコンポーネントのパスをlazy()関数に渡して、
その戻り値として、何かしらの変数に代入すること。

e.g.)
import { useState, lazy } from "react";

const LazyComponentA = lazy(() => import("./components/ComponentA"));
console.log(LazyComponentA)
const Example = () => {
  const [compA, setCompA] = useState(false);

  return (
    <>
      <button onClick={() => setCompA((prev) => !prev)}>ComponentA</button>
      {compA && <LazyComponentA />}
    </>
  );
};



## Suspenseコンポーネントを使って、loadingを表示させる。
Suspenseコンポーネントで、ローディングに時間のかかる要素に対して、待機画面を表示させる。
中に表示させておきたい内容はfallbackのpropsに対して渡す。
e.g.)

 return (
    <>
      <button onClick={() => setCompA((prev) => !prev)}>ComponentA</button>
      <Suspense fallback={<div>now loading</div>}>
        {compA && <LazyComponentA />}  
      </Suspense>
    </>
  );


##startTransition
ダイナミックインポートを非同期にバックグラウンドで取得するまで、その直前の画面を表示し続けてくれる。
<button onClick={() => {
        startTransition(() => {
          setCompA((prev) => !prev)}
        )}
      }
>
//onClickの時に無名のコールバック関数が呼ばれて、その中の処理として、
//startTransition()を呼び出しテイル。
//startTransitionの引数に取られた、コールバック関数内に記述された関数処理が、終了するまで、非同期に画面を更新しないでくれるってことですね。
