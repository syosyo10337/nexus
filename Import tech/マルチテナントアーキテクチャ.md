# マルチテナントアーキテクチャ

## データベース設計パターン：パフォーマンスと利便性のバランス

### ✅ **パターン1: Shared Database, Shared Schema (共有DB・共有スキーマ)**

すべてのテナントが同じデータベースとスキーマを共有し、各テーブルにテナントIDカラムを持つ方式です [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)。

**メリット:**

- 最もシンプルで設定・管理が容易 [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- 最高のリソース効率と簡素化された管理が可能 [SuperTokens](https://supertokens.com/blog/multi-tenant-architecture)

- 全体的な需要に基づいてリソースをスケールアップまたはスケールダウンできる [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- コスト効率が最も高い

**デメリット:**

- データ分離が弱く、**カスタマイズが限定的** [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- Noisy Neighbor問題：1つのテナントの過剰な負荷が他のテナントに影響 [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- クエリロジックが複雑になる：常にテナントIDをクエリに含める必要がある [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- 分離制御が失敗した場合のリスクが高い [SuperTokens](https://supertokens.com/blog/multi-tenant-architecture)

**適用ケース:**

- すべてのテナントがサイズと機能要件が類似している場合 [Medium](https://medium.com/mongodb/multi-tenancy-and-mongodb-5658512ed398)

- IoT系のユースケースで高いスケーラビリティが必要な場合 [Medium](https://medium.com/mongodb/multi-tenancy-and-mongodb-5658512ed398)

- 小規模なテナントが多数存在するSaaSプラットフォーム

**実装のベストプラクティス:**  
すべてのテーブルにテナントIDを非正規化して持たせることが推奨されます。データベース純粋主義者は反対するかもしれませんが、将来のスケーリングとデータの安全性が容易になります [Crunchy Data Blog](https://www.crunchydata.com/blog/designing-your-postgres-database-for-multi-tenancy)。

### ✅ **パターン2: Shared Database, Separate Schemas (共有DB・個別スキーマ)**

CHIMERはこれ

すべてのテナントがデータベースを共有しますが、各テナントが独自のスキーマを持つ方式です [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)。

**メリット:**

- データ分離とカスタマイズが向上 [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- 共有アプリケーションインスタンスによる効率的なリソース利用と、データベースレベルでのデータ分離を維持 [Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/saas-tenancy-app-design-patterns?view=azuresql)[GeeksforGeeks](https://www.geeksforgeeks.org/system-design/multi-tenancy-architecture-system-design/)

- バックアップやメンテナンス作業を1つのテナントに対して実行しても、他のテナントに影響を与えない [MongoDB](https://www.mongodb.com/docs/atlas/build-multi-tenant-arch/)

- MongoDB内蔵のRole-Based Access Control (RBAC)を使用してより良いセキュリティを実現 [Medium](https://medium.com/mongodb/multi-tenancy-and-mongodb-5658512ed398)

**デメリット:**

- 管理がより複雑 [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- アプリケーション層の複雑さが増加 [MongoDB](https://www.mongodb.com/docs/atlas/build-multi-tenant-arch/)

- スケーリングには限界がある：MongoDBでは1つのクラスタで推奨される最大コレクション数は10,000 [Medium](https://medium.com/mongodb/multi-tenancy-and-mongodb-5658512ed398)

- 単一クラスタ内での大規模化に問題が発生する可能性 [MongoDB](https://www.mongodb.com/docs/atlas/build-multi-tenant-arch/)

**適用ケース:**

- 中規模のテナント数（数百～数千）

- テナントごとにカスタマイズ要件がある場合

- セキュリティと分離のバランスが重要な場合

**MongoDBでの注意点:**  
各新しいコレクションとインデックスに必要な基礎データファイルが計算リソースを消費するため、スケーリングに制限があります [Medium](https://medium.com/mongodb/multi-tenancy-and-mongodb-5658512ed398)。

### ✅ **パターン3: Separate Databases (個別データベース / Database-per-Tenant)**

各テナントが専用のデータベースを持つ方式で、2025年現在、厳格なセキュリティとコンプライアンスが必要なエンタープライズSaaSアプリケーションのゴールドスタンダードとされています [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)[Lktechacademy](https://www.lktechacademy.com/2025/10/multi-tenant-saas-architecture-database-per-tenant.html)。

**メリット:**

- 最大限のデータ分離とセキュリティを提供 [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- データ漏洩のリスクが実質的にゼロ [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- パフォーマンス分離：テナントが互いのパフォーマンスに影響を与えない [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- カスタマイズが容易：テナントごとに異なるスキーマや設定が可能 [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- バックアップ/リストア、デバッグが簡単で、テナント間サポート問題が減少 [Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/saas-tenancy-app-design-patterns?view=azuresql)

- 規制要件への準拠が容易 [Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/saas-tenancy-app-design-patterns?view=azuresql)

**デメリット:**

- リソース集約的でコストが高い [daily.dev](https://daily.dev/blog/multi-tenant-database-design-patterns-2024)

- 管理が煩雑になる、特にテナント数が増加すると [Medium](https://medium.com/@seetharamugn/complete-guide-to-multi-tenant-architecture-d69b24b518d6)

- 複数のデータベースインスタンスのため、通常インフラストラクチャコストが高い [Lktechacademy](https://www.lktechacademy.com/2025/10/multi-tenant-saas-architecture-database-per-tenant.html)

- リソース割り当ての冗長性による高い運用コスト [Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/saas-tenancy-app-design-patterns?view=azuresql)

**適用ケース:**

- GDPR、HIPAAなどのコンプライアンスのための最大限のデータ分離が必要な場合 [Lktechacademy](https://www.lktechacademy.com/2025/10/multi-tenant-saas-architecture-database-per-tenant.html)

- 独自のデータベースを必要とするエンタープライズ顧客がいる場合 [Lktechacademy](https://www.lktechacademy.com/2025/10/multi-tenant-saas-architecture-database-per-tenant.html)

- Noisy Neighbor問題を回避する必要がある場合 [Lktechacademy](https://www.lktechacademy.com/2025/10/multi-tenant-saas-architecture-database-per-tenant.html)

- 規制対象業界（金融、医療など）

**スケーリングのベストプラクティス:**  
Elastic Poolを使用して、サブスクライバーテナント用の単一テナントデータベースをリソースプールに配置し、テナントあたりのデータベースコストを削減できます [Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/saas-tenancy-app-design-patterns?view=azuresql)。

同一インスタンス上に複数のdabaseを持つこともできそう。
