# ノート記述方針

## 核となる原則

メモの価値は「情報を保存すること」ではなく「自分の頭を通過させること」にある。きれいに写しても定着しない。受動的な記録ではなく、能動的な再構成だけがメモする価値を持つ。

**クローズドブック原則**: ソースを閉じてから書く。書けなければ理解が足りていない証拠。戻って読み直す。これがすべての土台。

## 書くかどうかの判断基準

| レベル | 判断 | 例 |
| --- | --- | --- |
| 自分にとって自明 | **書かない** | — |
| 理解してるが手順・構文を忘れる | **ポインタだけ** | 「○○本 p.XX」やURLを一行 |
| 概念は知ってるが毎回曖昧になる | **自分の言葉で書く** | ここが主戦場 |

同じ本を読んでもメモすべき箇所は人によって違う。「あれ、どっちだっけ？」と毎回なるポイントこそ対象。頻度が増えること自体は問題ない。

## 場面 1: インプット学習（本・記事・動画）

### やること

- **マルチパス**で読む。1回目は全体像、2回目でコード実行、3回目で自分の言葉に変換
- セクションを読み終えたらソースを閉じて要約を書く（クローズドブック要約）
- 「なぜそうなるのか？」「どう実装されているのか？」と精緻化質問を自分に投げる
- 悪い例: 「Repeatable Read は dirty read を防ぐ」（コピー）
- 良い例: 「なぜ防げるのか？ → 読み取り時にスナップショットを固定するから。Read Committed との違いは…」

### やらないこと

- ソースを開いたまま文章を整形してコピーする（理解の錯覚）
- 検索すれば出てくる定義・分類・比較表をそのまま転記する
- ハイライト + 引用のコレクション

### Framework 学習 vs 深いトピック

| | Framework（React, K8s ツール等） | ディープトピック（OS, 分散システム等） |
| --- | --- | --- |
| ノートの形 | チートシート、スニペット、落とし穴メモ | 自分の言葉による概念の要約 |
| 定着方法 | すぐプロジェクトで使う | 精緻化質問、人に教える |
| 書く基準 | ポインタで十分なことが多い | 自分の言葉で書く主戦場 |

## 場面 2: 実装中の発見（デバッグ・詰まり・気づき）

### なぜ別扱いするのか

実装中の「詰まり」には独自の価値がある。具体的な文脈があり、自分の理解のギャップが明確に露呈し、再発する可能性が高い。教科書では学べない現場の知識はここから生まれる。

### TIL（Today I Learned）

小さな発見を低摩擦でキャプチャする。意図的に軽量に。

- 数文 + コード例、200語以下
- 「ブログ記事にするほどではないが、忘れたくない」ものに最適
- ハードルの低さが特徴。10〜15分で書ける

### デバッグログ

詰まった時の思考過程をキャプチャする。最も価値があるのは**自分の誤解が何だったか**。

書くべき4要素:

1. **何をしようとした** → 目的
2. **何が起きた** → 症状
3. **なぜダメだった** → 原因（自分の誤解を含む）
4. **どう解決した** → 解法

間違いを削除しない。学習履歴が保存される。

### 汎化された例

実装中に学んだことを、プロジェクト固有の詳細を取り除いた汎用版として書く。理論（本/ドキュメント）と実践（チケット/PR）の両方にリンクすることで、転用可能な知識になる。

## ノートの種類

| 種類 | 性質 | 書き方 | 例 |
| --- | --- | --- | --- |
| **概念ノート** | 理解が深まると進化する | 自分の言葉で。精緻化質問を含む | カーネルとは、CAP定理 |
| **スニペット** | 安定したルックアップ | コマンド + いつ使うか + 簡単な説明 | git の特定操作 |
| **デバッグログ** | 誤解→修正の記録 | 4要素（目的・症状・原因・解法） | 認証が通らなかった時 |
| **ポインタ** | 情報の在処 | URL or ページ番号だけ | 公式ドキュメント参照 |

概念ノートとスニペットは本質的に異なる。概念ノートはリンクと精緻化で価値が増す。スニペットは安定した参照先として機能する。
