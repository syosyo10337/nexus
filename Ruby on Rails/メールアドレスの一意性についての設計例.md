# ** [メールアドレスの一意性についての設計例]

---

==(大文字小文字の区別をどう扱うか？)==

1. validationレベルでの一意性の確保(caseの区別)

2. databaseレベルでの一意性の確保

3. before_saveフィルターによりデータベースに保存されるとき小文字に強制する。

**step.1.フォーマットを自作した変数(VALID_EMAIL_REGEX)とは別に、uniquenessをつける。**

---

```Ruby
#例
validates :email, presence: true, 
									length: { maximum: 255},
                  format: { with: VALID_EMAIL_REGEX },
                  uniqueness: { case_sensitive: false }
```

rails デフォルトだと、case_sensitive : true(大文字小文字は区別される)なので、区別しない上で一意性を持つvalidationをかける

==!!ここでの問題点!! --データベースに保存する時に一意であることは確約されたが、データベース内での一意性は保証されていない。(つまり、データベースにデータが保存される直前も直前に、重複ユーザがvalidationを通過する可能性があるってこと。(送信ボタンの複数クリックが要因で)==

**step.2 データベースレベルでの一意性を、****検索のことも考慮し****、indexと共に付与する。**

---

```Ruby

#migrationファイルを作成して
#例
class AddIndexToUsersEmail < ActiveRecord::Migration[6.0]
  def change
    add_index :users, :email, unique: true
  end


#$ rails db:migrateを忘れずに
```

**step.3 データベースによって、大文字小文字の区別がされる場合もあるので、データベースの仕様に依存しないために,小文字に変換してsaveする**

---

```Ruby
class User < ApplicationRecord
  before_save { self.email = email.downcase }
# before_save { emai.downcase! }でも良い。

#中略

end




#リファクタリング。コールバック(before_save)では、:メソッド名という形でメソッド参照ができるので、privateなメソッドを定義して、メソッド参照に書き変える.

@「app/models/users」

before_save :email_downcase
.
.
private
	def email_downcase
		email.downcase!
	end
```

**補足)**deviseでは、case_sensitive: trueのまま、validationを評価する直前(before_validation)で小文字に変更することで、validationレベルとデータベースレベルでの大文字小文字区別の問題を解消している。
