# N + 1 問題

---

アプリケーションからSQLを発行する際に起こるパフォーマンス上の問題の一つ。

具体的には、

```Ruby
class User < ActiveRecord::Base
	has_many :cats,
end

class Cat < ActiveRecord::Base
	belongs_to :user
end
```

上のような関連づけされたモデルがある時に、

全飼い主の猫一覧を表示させたい場合について考える

```Ruby
#事前にユーザをすべて取得しておき
@users = User.all


#viewにおいて
@users.each do |owner|
	owner.cats.each do |cat|
		cat.name
	end
end
```

とすることで、表示させる。

しかし、

この時,親テーブル(1対多の1側)へのレコードアクセス1回に対して、関連する子テーブル(1対多の多側)へのレコードアクセスが数回、つまりN回発行される。

```Ruby
#@usersへUser.allの返り値を格納する時
SELECT `users`.* FROM `users`

#viewにて、それぞれのユーザの
SELECT `cats`.* FROM `cats`  WHERE `cats`.`user_id` = 1
SELECT `cats`.* FROM `cats`  WHERE `cats`.`user_id` = 2
SELECT `cats`.* FROM `cats`  WHERE `cats`.`user_id` = 3
SELECT `cats`.* FROM `cats`  WHERE `cats`.`user_id` = 4
```

**このように、あるテーブルへの1回のレコードに対して、そのテーブルに関連するテーブルへのアクセスがN回付随することを、N+1問題という。**

==※子テーブルへのレコードアクセス1回に対しても、親テーブルへのレコードアクセスがN回発生するケースもある。==　

## Railsにおける対応策

---

### includesメソッド

---

includesメソッドとは、引数に関連付け名を取り、関連付けられたテーブルに関するレコード情報も一度にまとめて取得し、ActiveRecord＿Relationインスタンスとして保持させる。

==※関連付け名は必ずしも、テーブル名にはならない==

```Ruby
ex)
@users = User.includes(:items)

#これによって発行されるSQL
User Load (0.2ms)  SELECT "users".* FROM "users"
Item Load (0.4ms)  SELECT "items".* FROM "items" 
									 WHERE "items"."user_id" 
									 IN (?, ?)  [["user_id", 1], ["user_id", 2]]

@users.class
=>User::ActiveRecord_Relation
```

上のように、@users変数に、関連のあるitemsに関する情報も取得していると

```Ruby
#viewにおいて
@users.each do |owner|
	owner.items.each do |item|
		puts item.name
	end
end
```

@users変数にはすでにクエリが発行され、itemsテーブルに関する情報も入っているので、SQLはそれ以上発行されることなく、すべてのユーザが持つ、アイテム一覧を取得することができる。
