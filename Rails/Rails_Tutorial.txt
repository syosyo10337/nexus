app/	--モデル、ビュー、コントローラ、ヘルパーなどを含む主要なアプリケーションコード

doc/	マニュアルなど、アプリケーションのドキュメント
lib/	ライブラリやモジュール置き場
log/	アプリケーションのログファイル
public/	エラーページなど、一般（Webブラウザなど）に直接公開するデータ
bin/rails	コード生成、コンソールの起動、ローカルのWebサーバーの立ち上げなどで使うRailsスクリプト
test/	アプリケーションのテスト
tmp/	一時ファイル
README.md	アプリケーションの簡単な説明
Gemfile	このアプリケーションに必要なGemの定義ファイル。bundle installをする時これを参照している。
Gemfile.lock	アプリケーションで使われるgemのバージョンを確認するためのリスト
config.ru	Rackミドルウェア用の設定ファイル
.gitignore	Gitに取り込みたくないファイルを指定するためのパターン

$ rails c(onsole)	--railsアプリの環境を反映したconsoleの立ち上げ(opt: --sandbox (砂場)環境(=仮想環境)で実行できる。)

****[MVC(Model/View/Controller)モデル] ==============================================
Railsのアーキテクチャ。Webサーバーにリクエスト（request）を送信され、リクエストを処理する役割のRailsのコントローラ（controller）に渡される。コントローラは、静的なサイトではすぐにビュー（view）を生成してHTMLをブラウザに送り返す。動的なサイトでは、コントローラは（ユーザーなどの）サイトの要素を表しており、データベースとの通信を担当しているRubyのオブジェクトのモデル（model）と対話します。モデルを呼び出した後、コントローラは、ビューを描画し、完成したWebページをHTMLとしてブラウザに返す。



***[View]-----------------------------------------
--見た目を構成するファイル(.html.erb) 
*ERB --(Embedded Ruby)RubyのスクリプトをHTMLなどに埋め込むためのライブラリ(テンプレートシステム)。<% %>で囲んだ部分にRubyのコードを書く。<%= %>は評価された値を出力する。
<%# %>でコメントアウトする。

**手動でviewを有効化する手順
①ルーティングを書く(アクションとURLを対応させる。)
②アクションをコントローラー内に記述する。
③template(view)ファイルを新規作成。


**Helper --RailsにおけるへルパーとはViewをDRYに作るためのモジュール。モデルやコントローラでも利用できるが、そもそもはViewは使うために定義されている。(組み込みヘルパーの一例:link_to/form_with/label_for/image_tag/render)そのため、viewで,カスタムヘルパーを使う際、ヘルパーの親クラスであるApplicationHelperが自動ですべてのViewファイルにincludeしてくれている。モデル/コントローラで使用する際にはinclude必須です。
+カスタムヘルパー(新しく自ら作成したメソッド)helperのファイルに記述する。->すべてのviewで利用可能。


**viewなどのファイルでURLを参照したい時に、生のURLをハードコードする方法と、名前付きルート(Named Route)を使う方法がある。

**名前付きルート(Named Routes) -- _path/_urlで終わる形を取り、URLを指定できるヘルパー(メソッド)。
+_urlと_pathの違い
ex)
about_path => '/about'  #_pathだと"/"以下を返す
about_url  => 'https://www.example.com/about'	#_urlだと、完全なURLを返す

*慣習に従い、基本的には_pathを使い、redirectの場合のみ_url書式を使う。これは、HTTPプロトコルに忠実考えた時、リダイレクトには完全なURLを求められることに起因します。ただ、実際のブラウザでは_path書式にも対応している場合が多いです。

ex1)'/users/1'に対してGETメソッドでHTTPリクエストが送られた 「params[:id] => 1」

ex2)*フォーム送信がされた時のparamsの中身一例
params => {:user => { :name => "Foo Bar",		#ネストしたハッシュ(hash of hash)になってるね。
		:email => "foo@invalid",
		:password => "[FILTERED]",
	:password_confirmation => "[FILTERED]"},
	:controller => xxx,
	:action => xxx,
	.
	.	
}


**flash --中身がリクエストごとにクリアされる変数。つまり、直後のリクエストで1度だけ表示される(サクセスメッセージなど利用)、任意のキーに対して、文字列を代入しておき、リダイレクト後に表示させる。慣習的に:successなどが使われる。(自由に設定可能)

*名前付きルート --prefix_path/_urlの形で,パス名を変数として扱える。
これはurlヘルパーとも呼ばれている。これは、route.rbの記述をもとにパスを作成するものなので、バックエンドでフロントで用いるURLなどが欲しい場合に、Controllerやviewが存在しない場合でも、route.rbの記述と適切なurlヘルパーを使用することで、URLを生成することはできる。ただし、
```
include Rails.application.routes.url_helpers
```
をすると、エラーが発生した事例もあるのと、本来の用途とは異なるため、非推奨な対応になる。

**{アセットパイプライン(Asset Pipeline) } ----------
-CSS、JavaScript(Webpack,Yarn)、画像などの静的コンテンツを最小化(不要な空白やインデントを取り除く)や圧縮して連結する機能を提供するフレームワーク。(開発者にわかりやすいように、特定の場所にファイルを分割して管理したものを、読み込む際には1つのファイルとしてブラウザに出力できるようにしている仕組み)

{メリット}
開発環境で、scssなどのスタイルシートをアセットディレクトリに分けて管理して、インデントしまくって管理してたけども、本番環境としては、ロードに時間がかかるので非効率。それを解決するために、Asset Pipelineがすべてのスタイルシートを1つのCSSファイル（application.css）にまとめ、インデントを消してファイルサイズを最小化までしてくれます。これによってユーザ体験の向上につながる(クライアント視点だと1つのcssファイルを読み込んでいるだけ)。


①アセットディレクトリ --静的ファイルを目的別にグループ化して保存しておく。(custom.scssはアプリ固有のアセットだよね？だからapp/assetsに入れたよ)
app/assets: 現在のアプリケーション固有のアセット
lib/assets: あなたの開発チームによって作成されたライブラリ用のアセット
vendor/assets: サードパーティのアセット（デフォルトでは存在しません）

②マニュフェストファイル --それぞれのアセットディクレトリ内で、ファイルのまとめ方を記述しているファイル。
(実際にアセットをまとめる処理を行うのはSprocketsというgem) 。また、マニフェストファイルはCSSとJavaScritには適用されますが、画像ファイルには適用されません

③プリプロセッサエンジン --①、②で見たようにディレクトリにアセット(cssや、jsや画像の資産)を配置し、マニフェストて統一した後に、ブラウザに配信できるよう結合する役割をするエンジン。
Railsはどのファイルに対してどのプリプロセッサを使うのか、ファイル名の拡張子を使って判断する。ex)拡張子が.erbであれば埋め込みRuby（ERb）用の、scssであればSass用の前処理（プリプロセス）を実行
ex)foobar.js.erb というようにすると、右から左に埋め込みルビーが実行され、その後それをjsのコードとして処理するというようにつなげることもできる。

***{Migration}-------------------------
*マイグレーションファイル --データベースを生成する際の設計図(変更内容はchangeメソッド内に記述される)。マイグレーションファイルを実行することで、記述した内容に基づいてインクリメンタルに、データテーブルが生成/更新されます。(データベースの定義をRubyで記述できるので,sql不要。DBの隠蔽を可能している。)

*マイグレーションエラー --データベースに反映されていないマイグレーションファイルが存在する状態で、どこかのページにアクセスすると発生するエラー。マイグレーションファイルを作成したら必ず、rails db:migrateすること

*$ rails db:migrate:status --マイグレーションファイルの状態が表示される。upになっていれば実行済み

*$ rails db:rollback --データベースを1つ前のマイグレーションの状態に戻す。
*$ 
0 --データベースを最初の状態に戻す(ver0に戻す、)#?また、数字を指定すればその時の状態に戻せる)

**{簡易的なデータモデルの関連付け(Associations)} -------------------------
has_many/belongs_toなどでデータモデル同士を関連づけること。それぞれのモデルで定義する。
+dependentなどの依存性の追加/外部キー制約(foreign_key constraint)もできる。
ex)
＠「app//user.rb」
has_many :microposts, dependent: :destroy

@「app/models/micropost.rb」
belongs_to :user

***#Railsの仕様として、参照先のモデル名(クラス名)+_idというsuffix(接尾辞)を形を取るカラム名を外部キーとして使うことを前提(デフォルト)としている

***belongs_to --関連付けで指定するモデル名(関連付け名)は必ず「単数形」。他のモデルのインスタンスに従属する。(相手を参照先にする。)また、自身にリンクカラムを持つ。(-> 自身がが外部キーを持つテーブルである)


***has_many --関連付けを宣言する際、相手のモデル名は「複数形」で指定する。指定した相対側のモデルが自身への参照してくる(つまり自信が相手にとっての参照先となる。)。相手側(参照する側)がリンクするカラム持っている。
(内部的には、has_manyの引数(複数形)をclassifyメソッドを用いてクラス名に変換している。) 
ex)
has_many :microposts
#=> Micropost


**has_many :through --他方のモデルと「多対多」のつながりを設定する場合によく使われます。、2つのモデルの間に「第3のモデル」（joinモデル）が介在し、それによって相手のモデルの「0個以上」のインスタンスとマッチします。
ex)class Physician < ApplicationRecord
has_many :appointments
has_many :patients, through: :appointments
end

class Appointment < ApplicationRecord
	belongs_to :physician
	belongs_to :patient
end

class Patient < ApplicationRecord
	has_many :appointments
	has_many :physicians, through: :appointments
end

**{ActiveStorage}-----------------------
Amazon S3などのクラウドストレージサービスへのファイルのアップロードや、ファイルをActive Recordオブジェクトにアタッチする機能を提供する。画像だけでなくさまざまなバイナリファイルを扱える。

*ActiveStorageの導入
$ rails active_storage:install
#添付ファイルの保存に用いるデータモデル作成のマイグレーションが1つ生成されるので
$ rails db:migrate


**{ActiveStorageAPIメソッドの一例}
*has_one_attached --指定のモデルと、アップロードされた1つのファイルを関連付けるのに使います*has_many_attached 

ex)@「app/models/micropost.rb」

has_one_attached :image
#micropostモデルが、1つの:image(画像ファイル)を持つ


*attachメソッド。既存のデータモデル(例えばユーザのプロフ画像の更新)を行う時に,使う(update的なニュアンス)attached?メソッドも使えるようになっている。

**{Active Storageバリデーション用のgem}
ネイティブでサポートされていないので、
＠「Gemfile」
gem 'active_storage_validations'

(DBでのインデックスの役割と意味/応用的なAssociations)


**Railsは外部キーのカラムを自動的に作成することはありません。外部キーを使うには、マイグレーションで明示的に定義する必要があります。(gスクリプトのreferencesを使うとある種暗示的に作成される点に気を付ける)

****{もっとAssociations(belongs_to/has_many/:through)}--------

関連付けっていうとモデル同士の関係性を定義する(level1)
データベース上で外部キー主キーで関連させている(level2)
関連付けた名前をメソッドのようにしてRails上で、CRUDしている(level3)

belongs_to/has_many --macroメソッドを生成するメソッドこれらによって、
user.microposts.buildみたいのが実現している。
#引数に取るのはあくまで、関連付け名(:association name)であり、必ずしもモデル名や外部キーではない。理解としては、これから生成するメソッドの命名部だという認識が正確かと。

***{has_manyの使い方} ----
ex1)
class User < ApplicationRecord
	has_many :microposts, dependent: :destroy
end

User has_many microposts と言われると、Railsは
①データベースレベルの話(prim.keyとfore.keyの関連付けという面では)、、、、
:microposts =>Micropostクラス(モデル)を探して、
②その中で、User => user_id というカラム名をforeign_keyとして設定する。

③Ruby上での使えるようになるメソッドとしては、
user = User.firstと仮に定義すると、
user.microposts.buildや
user.microposts --インスタンスの持つ投稿一覧(インスタンスに従属するオブジェクトのcollection)
逆に(belongs_toで双方向的に関連づいているとき)
micropost = user.microposts.firstと定義すると
micropost.user -投稿が従属するユーザを表示
というように使える。

ex2)
class User < ApplicationRecord
	has_many :active_relationships, class_name: "Relationship",
	foreign_key: "follower_id",
	dependent: :destroy	
end
①	
デフォルトで行くと、
:active_relationships -> ActiveRelationshipクラス(モデル)を探すが、
extension(拡張)として、class_name: "Relationship"があるので、
#=>Relationshipクラス(モデル)を探す。

②その探したクラスの中で、User -> user_idカラムを外部キーとしてUserモデルのテーブル上でいう主キーを関連づける。
dependentはactive_relationshipsはユーザに依存するということ。

③使えるようになるメソッドとしては、
user = User.firstと仮に定義すると、
user.active_relationships.build(follwed_id: "xx")や
user.active_relationships --インスタンスの持つ関係一覧(インスタンスに従属するオブジェクトの集合などが使えるようになる。


***{belongs_toの使い方}---
ex1)
class Micropost < ApplicationRecord
	belongs_to :user
end
Micropost belongs_to :userと言われると、
①:user -> Userクラス(モデル)を探し、
② 大前提としてbelongs_toの主語側となるデータモデルに外部キーとなるカラムを持っておく必要があり、belongs_to :user -> user_idカラムをMicropostデータモデルから探し出して、外部キーとしたものをMicropostのuser_idとUser id(prim.key)が関連づける
③Ruby上では、
first_post = Micropost.firstと定義した時に
first_post.user #=>ファーストポストと関連づいたユーザインスタンスの情報を取得できる
#これは、見かけ上は、インスタンス.モデル名ではあるが、インスタンス.関連付け名-> belongs_to :hogeのhogeの部分であることを留意


ex2)
class Relationship < ApplicationRecord
	belongs_to :follower, class_name "User"
end
①:follower -> Followerクラスを探すが、オプションとしてclass_name "User"があるため、関連付けの相手クラスとしてはUserだと伝える。
②Relationshipクラス側のキー(つまり、外部キーは),素直にbelongs_to :follower -> follower_id(in Relationship)と、Userのprim.keyを関連づける。
③Ruby上では、
first_relationship = Relationship.firstと定義した時に
first_relationship.follower #=> follower_idと一致するユーザを取得する
#データベース上の一番最初に生成された関係の中でもフォローした側のユーザが返ってくる。(実用上の効能はあまりない)



***has_many :through --多対多の関連を定義する時に使うメソッド(とオプション)。中間テーブルを持たせたDB設計を行う際に持ちいる。
引数にとった関連付け名(単数形)_idとなる外部キーを :throughで指定されたassociationが意味するテーブル(モデル)にあることを確認して、その外部キーに関連する集合を構築する。(:class_name, :primary_key and :foreign_keyは、:throughに指定したassociationのものをそのまま使うため、無効になる。)
class User < ApplicationRecord
	
	has_many :following, 
	through: :active_relationships, source: :followed
end

①Railsはまず、:following -> following(単)_idの外部キーに関連した集合(参照先のデータ)を取得しようとするが、 source: :followedによって、followed_idを外部キーとした参照先の集合をとることと認識を改める。
②どのテーブルの外部キーであるかをthrough: :active_relationshipsから、関連付け名active_relationships(relationshipsテーブル)だと認識する


#(has_manyの内部的な話)
#基本的には、渡された引数をclassifyメソッドでクラス名(モデル)化して、関連付けをしようと試みる。また、自身のクラス名をunderscoreメソッドによって、モデル名_idという外部キーと認識されるカラムを相手のモデルから探して、関連付けをする。

*****{ルーティング応用}---------
メンバールーティング --user/1/xxだとidがついてそれぞれを認識するルーティング ref)collection

実装--リソースルーティングのブロックとしてメンバーブロックを追加する。

ex)
resources :users do
	member do
		get :following, :followers
	end
end

#上の例では
/users/:id/followingへGETリクエストが来た時に ->Users＃followingへ
/users/:id/followersへGETリクエストが来た時に ->Users＃followersへ
ちなみに名前付きルートはそれぞれ
following_user_path(:id)
followers_user_path(:id)で対応するようになる。
