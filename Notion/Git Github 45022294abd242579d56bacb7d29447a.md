# Git

---

- **[基本的なワークフロー]**
    
    1. ファイルの変更をindexへ追加。
    
    2. その変更をlocal repositoryにcommit
    
    3. 最後にコミットした内容をremote repositoryにpushする
    

- **[用語]**
    
    - **index(/staging area)  
        **ファイルの変更のsnapshot(ある時間における状態)を保存(store)して置く場所のこと。
    
    - **commit  
        **indexに追加した変更を、メッセージをつけてリポジトリに記録すること。
    
    - **(****==tips==****)コミットメッセージの書き方のコツ**
    
    - **conflict  
        **あるファイルについての変更の差分が競合する際に、Gitがどちらの履歴を選択すべきが選べずに、自動的にmergeできない状態のこと。
    
    - **コンフリクトが起きたら**
        
        1. git diffで差分を確認し、
        
        2. 手動で該当ファイルを編集する。
        
        3. git add /git commitでコミットすることで  
            mergeコミットが作成される。

# Github

---

- **[pull requestの開発フロー]**
    
    1. (developer) 作業対象のソース(remote repo)を clone または pull します。
    
    2. (developer) 作業用のブランチを作成し,開発作業をおこなう
    
    3. (developer) 作業が完了したらpushします。
    
    4. (developer) プルリクエストを作成します。
    
    5. (reviewer) 通知されたプルリクエストから変更を確認しレビューします。
    
    6. (reviewer) レビュー結果を判断し、必要ならば開発者にフィードバックします。
    
    7. (reviewer) レビューの結果、問題がない場合はマージします。
    
    8. (reviewer) レビューの結果、対応自体が不要となるなど、プルリクエスト自体が必要ない場合はクローズします。
    
    ==3〜7の工程を、必要な分だけ繰り返します==
