# EC2と関連コスト

# SSHの多段接続で詰まった話

---

# 問題

publicなサブネットにあるbastion severから、privateなwebサーバーに多段接続したい時に、応答がなく、接続できなかった。

ちなみに、bastion → webへの（ICMP有効にしてpingコマンド)は通じたので、インスタンス自体は死んでない。fingerprintにも問題があるのか？と疑ったが、リモート側が新規なので、変更かかったわけでもなく、違うだろう。。。。

## 原因

**セキュリティグループの設定が問題でした。**

**自作したSSH:22ポートでの通信を0.0.0.0/0から受け取る設定”だけ”をbastionサーバーにしていた。**

ここでいうレベルのセキュリティグループの設定は、パケットフィルタリングのこと。つまり当該のインスタンスがどこからの何の通信は通すのか？という設定するところ。

これの何が問題だったかというと、、、

AWSのVPCを作成した時にデフォルトであるdefaultのセキュリティグループの意味は、

[![](AWS/Attachments/Screen_Shot_2022-11-06_at_23.11.53.png)](EC2%E3%81%A8%E9%96%A2%E9%80%A3%E3%82%B3%E3%82%B9%E3%83%88/Screen_Shot_2022-11-06_at_23.11.53.png)

同じセキュリティグループ割り当てされたリソースからのインバウンドを許可する！という設定です。なので、

bastionサーバーを立てる時は、defaultと自作SSH:22をインバウンドに設定すること。そうしないと、同じくdefaultのセキュリティグループに所属しているprivateのwebサーバは、ソース(送信元)は、同じセキュリティグループを持つインスタンスからのアクセスしか受け付けない！！。ということ。

**まとめると、bastionサーバーに対してdefaultで定めたパケットフィルタリングの制限が必要だったわけではなく、webサーバー側からインバウンドの相手として認めてもらうために、defaultに設定する必要があった。ということ。**

==****Elastic IP**== —設定すると課金されます。　

デフォルトではインスタンス起動ごとに、動的にパブリックIPが割り当てられる。が、この機能を使うことで、パブリックIPを固定することができる

また、NATゲートウィを作成すると自動的に作成されるので注意が必要.

また、Route53を利用することで、AWS経由でカスタムドメインの取得が可能になる。
