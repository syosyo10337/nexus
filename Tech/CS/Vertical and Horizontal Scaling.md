# Vertical Scaling と Horizontal Scaling

## 概要

スケーリング（Scaling）は、システムの負荷に対応するためにリソースを増やす手法です。主に2つのアプローチがあります：

- **Vertical Scaling（垂直スケーリング / Scale-up）**: 単一ノードのリソースを増やす
- **Horizontal Scaling（水平スケーリング / Scale-out）**: ノード/マシン/サーバーの数を増やす

## Vertical Scaling（垂直スケーリング / Scale-up）

### Vertical Scalingの定義

単一のノード（サーバー）のリソースを増やすことで、システムの処理能力を向上させる手法です。

**具体例：**

- CPUコア数の増加
- メモリ（RAM）の増加
- ストレージの高速化・容量増加
- ネットワーク帯域の拡張

### Vertical Scalingの特徴

| メリット ✅ | デメリット ⚠️ |
| --- | --- |
| **初期の複雑さが低い**: 単一ノードの管理で済むため、運用が比較的シンプル | **単一障害点（SPOF）**: 1つのノードが故障するとシステム全体が停止するリスク |
| **迅速な性能向上**: リソースを増やすだけで即座に性能が向上 | **ハードウェアの限界**: CPU、メモリ、I/Oには物理的な上限がある |
| **予測可能な成長に適している**: 段階的な負荷増加に対応しやすい | **ダウンタイムが必要**: リソース追加時にシステム停止が必要な場合がある |
| **予算が限られている場合に有効**: 初期投資が比較的少ない場合がある | **コスト効率**: 高スケール時には単位性能あたりのコストが高くなる傾向 |

### Vertical Scalingが適しているケース

- **モノリシックアプリケーション**: 分散設計されていないレガシーシステム
- **シングルスレッド処理が中心**: 並列化が難しいワークロード
- **予測可能で緩やかな成長**: 急激な負荷増加が予想されない
- **迅速な性能向上が必要**: 短期的な性能ブーストが必要な場合

## Horizontal Scaling（水平スケーリング / Scale-out）

### Horizontal Scalingの定義

複数のノード（サーバー）を追加することで、ワークロードを分散させてシステムの処理能力を向上させる手法です。

**具体例：**

- サーバーインスタンスの追加
- ロードバランサーによる負荷分散
- データベースのレプリケーション・シャーディング
- マイクロサービスアーキテクチャでのサービス単位のスケーリング

### Horizontal Scalingの特徴

| メリット ✅ | デメリット ⚠️ |
| --- | --- |
| **高い可用性**: 1つのノードが故障しても他のノードで処理を継続可能 | **複雑性の増加**: オーケストレーション、ネットワーク、一貫性管理が必要 |
| **ほぼ無制限のスケーラビリティ**: クラウド環境ではほぼ線形にスケール可能 | **分散システムの課題**: データの一貫性、ネットワークレイテンシ、パーティション分割への対応が必要 |
| **ダウンタイムの最小化**: ノードの追加・削除を稼働中に行える | **初期コスト**: インフラストラクチャとツールのセットアップにコストがかかる |
| **コスト効率**: 大規模時には単位性能あたりのコストが低くなる傾向 | **ステートレス設計が必要**: 水平スケーリングを効果的に行うには、ステートレスな設計が推奨される |
| **地理的分散**: 複数のリージョンに配置することでレイテンシを改善可能 | |

### Horizontal Scalingが適しているケース

- **大規模で予測困難なワークロード**: 急激な負荷増加が予想される
- **高可用性が必須**: システムの可用性と耐障害性が重要
- **マイクロサービスアーキテクチャ**: 独立してスケール可能なサービス構成
- **ステートレスなアプリケーション**: セッション状態を共有しない設計
- **長期的な成長が見込まれる**: 将来的に単一マシンのリソース上限に達する可能性がある

## 比較表

| 特性 | Vertical Scaling | Horizontal Scaling |
| --- | --- | --- |
| **適したアーキテクチャ** | モノリシック、レガシー、シングルプロセッサ | 分散システム、マイクロサービス、ステートレスAPI |
| **耐障害性・可用性** | 低い（単一障害点） | 高い（フェイルオーバー可能） |
| **スケーリングの限界** | ハードウェアの物理的限界 | ほぼ無制限（クラウドでは線形スケール可能） |
| **複雑性とコスト** | 初期は低いが、高スケール時は高コスト | 初期は高いが、大規模時はコスト効率が良い |
| **ダウンタイム** | リソース追加時に必要 | 最小限（稼働中にノード追加可能） |

## ハイブリッドアプローチ（Diagonal Scaling）✅

**業界標準のベストプラクティス**: 垂直スケーリングと水平スケーリングを組み合わせる手法が推奨されています。

### 戦略

1. **初期段階**: Vertical Scalingで開始（シンプルな運用）
2. **成長段階**: ハードウェア限界に近づいたらHorizontal Scalingに移行
3. **並行運用**: 両方を組み合わせて最適化（例：ノードを垂直スケールしつつ、ノード数も増やす）

### 具体例

- **Kubernetes環境**: Vertical Pod Autoscaler（VPA）とHorizontal Pod Autoscaler（HPA）の併用
- **クラウド環境**: VMインスタンスのサイズを増やしつつ、インスタンス数も増やす
- **ML推論システム**: スパイク時は垂直スケールで対応し、負荷が安定したら水平スケールで追加ノードを配置

## ベストプラクティス（2024年時点）✅

1. **段階的なアプローチ**: 可能な限りVertical Scalingから開始し、ハードウェア限界に近づいたらHorizontal Scalingに移行
2. **ステートレス設計**: 可能な限りステートレスな設計を採用し、Horizontal Scalingを容易にする
3. **自動スケーリングツールの活用**: VPA、HPA、クラウドVMリサイザーなどの自動スケーリングツールを活用
4. **ボトルネックの監視**: CPU、メモリ、ディスクI/O、ネットワークレイテンシを継続的に監視
5. **高可用性の計画**: 垂直スケールシステムでもレプリケーションやバックアップが必要
6. **コスト最適化**: レアなピーク時に過剰な垂直スケールを避け、クラウドやサーバーレスで水平スケールを検討
7. **段階的な移行**: 垂直スケールは段階的に増加、水平スケールはフェーズごとにノードを追加し、分散システムの挙動を観察
