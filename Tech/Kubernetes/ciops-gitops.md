# CIOps vs GitOps

Kubernetes におけるデプロイ戦略として、伝統的な **CIOps** と、モダンな **GitOps** の 2 つのアプローチがあります。

## CIOps (Push 型)

CI ツール（GitHub Actions, CircleCI, Jenkins など）が主体となって、Kubernetes クラスタに対して変更を「プッシュ」する方式です。

### CIOps の仕組み

1. 開発者がコードを Git にプッシュする。
2. CI ツールがビルド・テストを実行する。
3. CI ツールが `kubectl apply` や `helm install` などを実行して、クラスタにマニフェストを反映する。

### CIOps のメリット

- **導入が容易**: 既存の CI パイプラインの延長で構築できるため、学習コストが低い。
- **直感的**: 「ビルドしてデプロイする」という流れがシンプルで分かりやすい。
- **柔軟性**: 命令的なコマンドを実行しやすいため、複雑なスクリプトによる制御が可能。

### CIOps のデメリット

- **セキュリティリスク**: CI ツールにクラスタへの書き込み権限（kubeconfig など）を持たせる必要があり、CI ツールが侵害された際の影響が大きい。
- **構成のドリフト（乖離）**: クラスタ側で手動変更が行われた場合、Git 上の定義と実際の状態がズレてしまう。
- **デプロイのブラックボックス化**: デプロイが成功したかどうかは CI ツール側のログを見ないと分からず、クラスタの状態を常に Git と同期し続ける仕組みがない。

## GitOps (Pull 型)

Git を「信頼できる唯一の情報源（Single Source of Truth）」とし、クラスタ側で動作するエージェント（Argo CD, Flux など）が Git の状態を「プル」して同期する方式です。

### GitOps の 4 つの原則

OpenGitOps (CNCF のワーキンググループ) では、GitOps を以下のように定義しています。

1. **宣言的 (Declarative)**
    システムの望ましい状態は「宣言的」に表現される必要があります。Kubernetes のマニフェスト（YAML/JSON）のように、手順ではなく「あるべき姿」を記述することを指します。
2. **バージョン管理と不変性 (Versioned and Immutable)**
    望ましい状態は、不変（Immutable）な形式で、バージョン管理が可能なストレージ（Git リポジトリ）に保存される必要があります。これにより、変更履歴が全て残り、以前の状態へのロールバックも容易になります。
3. **自動プル (Pulled Automatically)**
    宣言された「望ましい状態」は、ソフトウェアエージェントによってソースから自動的にプル（適用）される必要があります。
4. **継続的な調整 (Continuously Reconciled)**
    ソフトウェアエージェントは「実際の状態」を継続的に監視し、「望ましい状態」との差分がないかを確認し続けます。差分（ドリフト）がある場合は、自動的に修正（Reconciliation）を行います。

### GitOps の仕組み

1. 開発者がマニフェストファイルを Git にプッシュする。
2. クラスタ内の GitOps エージェントが Git の変更を検知する。
3. エージェントが Git の状態とクラスタの状態を比較し、差分があれば自動的に同期（Sync）する。

### GitOps のメリット

- **セキュリティの向上**: クラスタ内のエージェントが Git を見に行くため、外部（CI ツールなど）に強い権限を持たせる必要がない。
- **ドリフトの自動検知・修復**: クラスタが手動で書き換えられても、エージェントが自動的に Git の状態に引き戻してくれる。
- **運用の透明性**: 「Git にあるものが、クラスタにあるもの」という原則が守られるため、誰がいつ何を変更したかが Git の履歴ですべて追える。
- **迅速なロールバック**: `git revert` をするだけで、クラスタの状態も以前の状態に自動的に戻る。

### GitOps のデメリット

- **学習コスト**: Argo CD や Flux などの専用ツールの導入・運用知識が必要になる。
- **CI/CD の分離**: 「アプリのビルド（CI）」と「デプロイ（GitOps）」を切り離して考える必要があるため、設計が少し複雑になる。
- **シークレット管理**: マニフェストを Git で管理するため、パスワードなどの機密情報をどう扱うか（Sealed Secrets や外部 Secret Store との連携）を検討する必要がある。

## 比較まとめ

| 特徴 | CIOps (Push 型) | GitOps (Pull 型) |
| :--- | :--- | :--- |
| **デプロイ主体** | CI ツール (外部) | GitOps エージェント (内部) |
| **接続方向** | CI → Cluster | Cluster → Git |
| **セキュリティ** | CI に強い権限が必要 | 外部への権限露出が少ない |
| **ドリフト防止** | なし (定期実行が必要) | あり (常に監視・修復) |
| **主なツール** | GitHub Actions, Jenkins | Argo CD, Flux |
