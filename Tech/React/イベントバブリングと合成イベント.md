# イベントバブリングと合成イベント

## DOM のイベントバブリング（基本）

イベントバブリングとは、ある要素で発生したイベントが DOM ツリーを親方向に遡って伝播する仕組み。

```html
<div id="grandparent">
  <div id="parent">
    <button id="child">Click me</button>
  </div>
</div>
```

`#child` をクリックすると、イベントは以下の順序で伝播する:

1. `#child`（イベント発生元 = `e.target`）
2. `#parent`
3. `#grandparent`
4. `document`

この「泡のように下から上へ伝わる」動きがバブリング。`e.stopPropagation()` を呼ぶと、その時点で伝播が止まる。

```javascript
document.getElementById("parent").addEventListener("click", (e) => {
  e.stopPropagation(); // grandparent には伝わらない
});
```

## React の合成イベント（SyntheticEvent）

React は DOM のネイティブイベントをそのまま使わず、独自の**合成イベント（SyntheticEvent）**でラップしている。

### なぜ合成イベントが必要か

- **ブラウザ間の差異を吸収**: ネイティブイベントの挙動はブラウザごとに微妙に異なる。合成イベントが統一的なインターフェースを提供する
- **イベント委譲**: React は個々の DOM ノードにリスナーを登録せず、ルート（`#root`）に一括登録して委譲（delegation）する。これによりパフォーマンスが向上する

### 重要な違い: バブリングの経路

**DOM のネイティブイベント** → **DOM ツリー**を辿る
**React の合成イベント** → **React コンポーネントツリー**を辿る

通常はこの2つのツリーは一致するので問題にならない。しかし **Portal を使うとツリーが乖離**し、バグの原因になる。

## Portal とバブリングの罠

### Portal の特性

`createPortal` で描画された要素は:

- **DOM 上**: Portal 先（通常 `<body>` 直下）に配置される
- **React コンポーネントツリー上**: Portal を呼び出した元の位置に留まる

### 問題が起きるケース

テーブルの行にクリックハンドラがあり、行内のドロップダウンメニューが Portal で描画される場合:

```
DOM ツリー:
<body>
  ├── <table>
  │     └── <tr onClick={handleRowClick}>
  │           └── <td data-actions>
  │                 └── <EventActions />    ← Portal の呼び出し元
  │
  └── <div data-radix-portal>               ← DOM 上は完全に別の場所
        └── <div role="menu">
              └── <div role="menuitem">     ← ユーザーがクリック
```

DOM のネイティブイベントは `<div data-radix-portal>` → `<body>` と辿るため `<tr>` には到達しない。

```
React コンポーネントツリー:
<TableRow onClick={handleRowClick}>           ← 3. ここまで到達する！
  └── <TableCell data-actions>
        └── <EventActions>
              └── <DropdownMenuContent>       ← 2. React ツリーの親を辿る
                    └── <DropdownMenuItem>     ← 1. クリック発生
```

しかし React の合成イベントはコンポーネントツリーを辿るため、**Portal 内のクリックが `TableRow` の `onClick` まで伝播してしまう**。

### DOM チェックが効かない理由

`handleRowClick` 内でガード条件を書いても回避できないケースがある:

```typescript
const handleRowClick = (e: React.MouseEvent) => {
  const target = e.target as HTMLElement;

  // DOM ツリーを辿って data-actions を探す → Portal 内なので見つからない → null
  if (target.closest("[data-actions]")) return;

  // menuitem は <div> なので button にもマッチしない → null
  if (target.closest("button")) return;

  // ガードをすり抜けて実行されてしまう
  router.push(`/detail/${id}`);
};
```

`e.target` は実際の DOM 要素（Portal 内の menuitem）を返す。`closest()` は DOM ツリーを辿る。一方、イベント自体は React コンポーネントツリーで伝播している。**このミスマッチがバグの原因**。

### 解決策: stopPropagation

Portal のルート要素で `e.stopPropagation()` を呼ぶと、React の合成イベント伝播がそこで止まる:

```tsx
<DropdownMenuContent
  align="end"
  onClick={(e) => e.stopPropagation()}
>
  <DropdownMenuItem>公開</DropdownMenuItem>
  <DropdownMenuItem>削除</DropdownMenuItem>
</DropdownMenuContent>
```

```
React コンポーネントツリー:
<TableRow onClick={handleRowClick}>             ← 到達しない ✓
  └── <TableCell data-actions>
        └── <EventActions>
              └── <DropdownMenuContent          ← stopPropagation でここで停止
                    onClick={stopPropagation}>
                    └── <DropdownMenuItem>       ← クリック発生
```

## Radix UI の modal prop との組み合わせ

Radix UI の `DropdownMenu` には `modal` prop がある（デフォルト `true`）。

| 設定 | dismiss layer | フォーカストラップ | メニュー外クリック |
|---|---|---|---|
| `modal={true}`（デフォルト） | あり | あり | dismiss layer が吸収 → 下の要素に届かない |
| `modal={false}` | なし | なし | そのまま下の要素に到達する（click-through） |

`modal={false}` にすると、メニューを閉じるために外側をクリックした際に、その下にある要素（例: テーブルの行）のクリックイベントも発火してしまう。

### ベストプラクティス

2つの対策を組み合わせると堅牢:

| 操作 | 防御 |
|---|---|
| メニュー項目をクリック | `stopPropagation` で React 合成イベントの伝播を止める |
| メニュー外をクリックして閉じる | `modal={true}` の dismiss layer がクリックを吸収する |

## まとめ

- DOM イベントは **DOM ツリー**を辿り、React 合成イベントは **React コンポーネントツリー**を辿る
- 通常は一致するが、**Portal を使うと乖離**する
- Portal 内のクリックは DOM 上の親には届かないが、**React ツリー上の親には届く**
- `e.target.closest()` は DOM を辿るため、React ツリー経由の伝播を防ぐガードにならない
- Portal のルートで `e.stopPropagation()` を使うのが確実な解決策
