## DNSとは

**DNS (Domain Name System)** は、ドメイン名とIPアドレスの名前解決を行う、インターネットの階層的な分散型データベースシステムです。

ブラウザがインターネットリソースにアクセスできるよう、人間が読みやすいドメイン名（例：`www.example.com`）を、コンピュータが理解できるIPアドレス（例：`192.0.2.1`）に変換します。

## IPアドレスとは

TCP/IPの通信によるコンピュータネットワークに属する全てのデバイスには、一意に特定できる**IPアドレス**があります。このIPアドレスを使って、インターネット上の他のマシンがそのデバイスを特定します。

## ドメインの階層構造

ドメインは頻繁に変化するため、一括管理ができません。そこで階層構造で管理されています：

```text
ドメイン名 = ルートドメイン(.) + 下位のドメイン + ホスト名
例: www.lpi.or.jp.
```

### 階層の構成

1. **ルートドメイン (`.`)** - 最上位（通常は省略される）
2. **TLD (Top Level Domain)** - `jp`, `com`, `org` など
3. **SLD (Second Level Domain)** - `co`, `or`, `ne` など
4. **サブドメイン** - 組織やサービス固有の名前
5. **ホスト名** - `www`, `mail` など

各階層ごとにドメインを管理するデータベース（DNSサーバ）が世界中に分散して設置されています。

## 名前解決に必要な4つのサーバ

DNSの名前解決には、以下の4種類のサーバが連携して動作します：

### 1. DNSリカーシブリゾルバ（フルサービスリゾルバ）

- クライアント（Webブラウザなど）からのリクエストを受け取る
- DNS階層を横断してクエリを行う
- 通常、ISPや企業ネットワークが提供
- キャッシュ機能を持つ

### 2. ルートネームサーバ

- DNS階層の最上位に位置
- TLDネームサーバの場所（IPアドレス）を管理
- 世界中に13系統（A〜M）が分散配置

### 3. TLDネームサーバ(top level domain)

- `.com`, `.jp`, `.org` などのTLD情報を管理
- 各ドメインの権威ネームサーバの場所を保持

### 4. 権威ネームサーバ（権威DNSサーバ）

- 特定のドメインのDNSレコード情報を保持
- 最終的なIPアドレス情報を返す
- ドメインの所有者が管理

## ドメイン名がIPアドレスに変換されるプロセス

1. **クライアントからのリクエスト**
   - ブラウザが `www.example.com` にアクセスしようとする

2. **リカーシブリゾルバへの問い合わせ**
   - リゾルバのキャッシュを確認
   - キャッシュになければ次のステップへ

3. **ルートサーバへの問い合わせ**
   - `.com` のTLDサーバの場所を取得

4. **TLDサーバへの問い合わせ**
   - `example.com` の権威サーバの場所を取得

5. **権威サーバへの問い合わせ**
   - `www.example.com` のIPアドレスを取得

6. **クライアントへの返答**
   - 取得したIPアドレスをクライアントに返す
   - リゾルバでキャッシュする

## DNSのキャッシング

DNS問い合わせの効率化のため、複数のレベルでキャッシュが行われます：

### DNSサーバ自身のキャッシュ

- リカーシブリゾルバが過去の問い合わせ結果を保存
- TTL（Time To Live）に基づいて有効期限を管理

### ブラウザのDNSキャッシュ

- OSレベルやブラウザレベルでもキャッシュ
- 名前解決の高速化に貢献

## 主要なDNSレコードタイプ

権威ネームサーバが管理するDNSレコードの主な種類：

| レコード  | 説明                                       |
| --------- | ------------------------------------------ |
| **A** ✅  | ドメインのIPv4アドレスを指定（最も基本的） |
| **AAAA**  | ドメインのIPv6アドレスを指定               |
| **CNAME** | ドメインのエイリアス（別名）を設定         |
| **MX**    | メールサーバーを指定                       |
| **NS**    | ドメインのネームサーバーを指定             |
| **TXT**   | テキスト情報を格納（SPF、DKIM等で使用）    |
| **SOA**   | ゾーンの管理情報を格納                     |
| **PTR**   | 逆引き（IPアドレスからドメイン名）に使用   |
| **SRV**   | 特定サービスのポート情報を指定             |

## NS / A / CNAME の使い分け（実用ガイド）

DNS設定で最も頻繁に触る3つのレコードの関係と使い分け。

### NS（ネームサーバー）

「このドメインの DNS 管理者は誰か」を指定するレコード。ドメインレジストラ（ロリポップ、お名前.com 等）で設定する。

```text
soypoy.com の NS → dns1.p01.nsone.net（Netlify DNS）
soypoy.com の NS → xxx.ns.cloudflare.com（Cloudflare DNS）
```

NS を変更 = DNS の管理先を丸ごと切り替える。レコード（A, CNAME 等）は新しい DNS 側で再設定が必要。

### A レコード

ドメインを **IP アドレス（数字）** に直接変換する。

```text
soypoy.com → 192.168.1.1
```

- サーバーの IP が固定の場合に使う（EC2 Elastic IP、VPS 等）
- IP が変わるとレコードも手動で更新が必要

### CNAME

ドメインを **別のドメイン名** に変換する（エイリアス）。

```text
soypoy.com → soypoy-portal.netlify.app
                ↓（Netlify が内部で解決）
              実際の IP アドレス
```

- Netlify / Vercel / Heroku 等、IP が動的に変わるサービスでは CNAME が適切
- サービス側が IP を変更しても、CNAME なら自動で追従する

### 判断基準

| やりたいこと                         | 何を変える       | どこで設定する         |
| ------------------------------------ | ---------------- | ---------------------- |
| DNS 管理先を切り替えたい             | NS（委任レコード） | ドメインレジストラ     |
| 固定 IP のサーバーに向けたい         | A レコード       | DNS 管理画面           |
| PaaS（Netlify, Vercel 等）に向けたい | CNAME            | DNS 管理画面           |

> NS はレコードタイプとしては存在するが、自分の DNS 管理画面で編集するものではなく、レジストラ経由で親ゾーン（TLD サーバー）に登録される委任情報。

## DNSのゾーン

**ゾーン**とは、DNSの管理単位のことです：

- 特定のドメインとそのサブドメインをまとめて管理
- 権威サーバーがゾーンファイルとして情報を保持
- ゾーン転送により、プライマリ/セカンダリサーバー間で同期

## セキュリティとベストプラクティス

### DNSSEC ✅

- DNS情報の改ざんを防ぐ電子署名技術
- データの完全性と認証を保証

### 推奨される設定

- 複数のネームサーバーを設定（冗長化）
- 適切なTTL値の設定
- DNSキャッシュポイズニング対策
