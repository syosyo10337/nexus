スケジューラを持たないアプリで定期的なジョブの実行を行うにはcronを利用するのが自然。本では、cronと実行するjob(shell script)を同居させる2 プロセスのコンテナを作成する。

- cron（クーロン）とは
    
    ジョブ（スクリプト）を自動実行するためのデーモンプロセスです。そして、Linux システムの管理を行なう場合、ログのローテートや、バックアップなど、定期的に自動実行したいジョブが数多くあります。特に、バックアップなどは、システムへの負荷が大きいため、
    
    通常は、ユーザからのアクセスが少ない、深夜や早朝に行なわれます。
    
    このように、定期的に実行されるジョブは、crond というデーモンによって、自動的に行なうように管理することができます。そこで、今回は、cron の設定を行なうために必要なコマンドや、設定ファイルについて説明していきたいと思います。
    

## コンテナの構築の指針

> Each container should have only one concern

_Dockerのドキュメントより_

1つのコンテナに、１プロセスではなく、1つの関心事つまり、一つの役割(role)や問題領域(domain)のみにフォーカスされるべきという事。

また、webアプリの文脈で考えた時には、スケールアウトも加味して、コンテナ単位でレプリカとして複製された時に副作用なく、スタックできるか？という点についても考慮すると実践的です。

### ライブラリのダイナミックリンクの課題

---

static link libraryとは、必要なライブラリをアプリケーションに内包する形式で、ポータビリティに優れますが、容量はおおきくなります。

dynamic link libraryは、アプリケーションを実行する際にリンクするので、サイズは軽量になりますが、実行ホストに用意しておく必要があるライブラリを管理するのが大変です。

？？

このdynamic link libraryの問題は、Docker環境においても同様で、ADDやCOPYなどでホストからコンテナに対して、ファイルを注入するというユースケースは少なくありません。

また、CIを使ってアプリケーションを最終的にはDocker Imageとしてパッケージングするようなケースについて考えます。

CIの時間を短縮するためにも、CIのプロセス中に作成したアプリケーションをそのまま、コンテナ内にコピーして、コンテナごとのコンパイル時間を削減して、一度のコンパイルでアプリの配置をさせるという手法があります。この手法がコンテナ内でアプリをビルドするよりも手軽であると言えます。しかしこの手法にはトラップがあります。

つまり、CI側とコンテナ側での標準Cライブラリが違えば、CI側でコンパイルしたバイナリファイルはコンテナ側では動作しません。。。

ありがちなのは、CI側がglibc(GNU Cライブラリ)をダイナミックリンクするバイナリを作成し、コンテナ側がmusl(musl libcm・マッスル)を採用しているために動かないケースです。

解決策としては、全ての依存ライブラリをスタティックリンクするか、実行プラットフォーム上にglibcをインストールするかになるでしょう。

標準Cライブラリを都度CIサービスに応じて切り替えるのは効率的ではありません。そのため、Dockerコンテナ上での実行を想定したアプリケーションを作り際には、ネイティブライブラリを極力スタティックリンクしてビルドすることを第一に考えましょう。

# Docker Swarm

---

複数のホストに複数のコンテナを配置する際に用いるツールのことで、コンテナオーケストレーションシステムの一つです。Docker Swarmは複数のdockerホストを束ねてクラスタ化します。

## service

複数のホストと複数のコンテナにまたがるアプリケーションの特定のコンテナを制御するための仕組み

1つのイメージしか実行できない。が複製などもできる。

```Bash
# e.g.)managerという仮想dockerホスト上で、serviceをechoという名前で立てている。
$ docker container exec -it manager \
> docker service create --replicas 1 --publish 8000:8000 --name echo registry:5000/example/echo:latest
pgpoblmk4kz99n9afnh51zop7
```

## stack

一つのserviceでは1つのアプリケーションイメージしか扱うことができない。アプリ全体として複数のserviceが協調して成立するものもあります。これを解決するserviceの上位概念がstackと呼ばれるものになります。stackでは複数のserviceを扱うことができ、アプリケーション全体の構成を定義します。

==stackが扱うアプリケーションの粒度はCompose==を同じです。stackはswarm上でスケールインアウトできるようになったcomposeという位置づけ。
