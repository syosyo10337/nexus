## DNSとは

**DNS (Domain Name System)** は、ドメイン名とIPアドレスの名前解決を行う、インターネットの階層的な分散型データベースシステムです。

ブラウザがインターネットリソースにアクセスできるよう、人間が読みやすいドメイン名（例：`www.example.com`）を、コンピュータが理解できるIPアドレス（例：`192.0.2.1`）に変換します。

## IPアドレスとは

TCP/IPの通信によるコンピュータネットワークに属する全てのデバイスには、一意に特定できる**IPアドレス**があります。このIPアドレスを使って、インターネット上の他のマシンがそのデバイスを特定します。

## ドメインの階層構造

ドメインは頻繁に変化するため、一括管理ができません。そこで階層構造で管理されています：

```
ドメイン名 = ルートドメイン(.) + 下位のドメイン + ホスト名
例: www.lpi.or.jp.
```

### 階層の構成

1. **ルートドメイン (`.`)** - 最上位（通常は省略される）
2. **TLD (Top Level Domain)** - `jp`, `com`, `org` など
3. **SLD (Second Level Domain)** - `co`, `or`, `ne` など
4. **サブドメイン** - 組織やサービス固有の名前
5. **ホスト名** - `www`, `mail` など

各階層ごとにドメインを管理するデータベース（DNSサーバ）が世界中に分散して設置されています。

## 名前解決に必要な4つのサーバ

DNSの名前解決には、以下の4種類のサーバが連携して動作します：

### 1. DNSリカーシブリゾルバ（フルサービスリゾルバ）

- クライアント（Webブラウザなど）からのリクエストを受け取る
- DNS階層を横断してクエリを行う
- 通常、ISPや企業ネットワークが提供
- キャッシュ機能を持つ

### 2. ルートネームサーバ

- DNS階層の最上位に位置
- TLDネームサーバの場所（IPアドレス）を管理
- 世界中に13系統（A〜M）が分散配置

### 3. TLDネームサーバ(top level domain)

- `.com`, `.jp`, `.org` などのTLD情報を管理
- 各ドメインの権威ネームサーバの場所を保持

### 4. 権威ネームサーバ（権威DNSサーバ）

- 特定のドメインのDNSレコード情報を保持
- 最終的なIPアドレス情報を返す
- ドメインの所有者が管理

## ドメイン名がIPアドレスに変換されるプロセス

1. **クライアントからのリクエスト**
   - ブラウザが `www.example.com` にアクセスしようとする
   
2. **リカーシブリゾルバへの問い合わせ**
   - リゾルバのキャッシュを確認
   - キャッシュになければ次のステップへ

3. **ルートサーバへの問い合わせ**
   - `.com` のTLDサーバの場所を取得

4. **TLDサーバへの問い合わせ**
   - `example.com` の権威サーバの場所を取得

5. **権威サーバへの問い合わせ**
   - `www.example.com` のIPアドレスを取得

6. **クライアントへの返答**
   - 取得したIPアドレスをクライアントに返す
   - リゾルバでキャッシュする

## DNSのキャッシング

DNS問い合わせの効率化のため、複数のレベルでキャッシュが行われます：

### DNSサーバ自身のキャッシュ
- リカーシブリゾルバが過去の問い合わせ結果を保存
- TTL（Time To Live）に基づいて有効期限を管理

### ブラウザのDNSキャッシュ
- OSレベルやブラウザレベルでもキャッシュ
- 名前解決の高速化に貢献

## 主要なDNSレコードタイプ

権威ネームサーバが管理するDNSレコードの主な種類：

| レコード | 説明 |
|---------|------|
| **A** ✅ | ドメインのIPv4アドレスを指定（最も基本的） |
| **AAAA** | ドメインのIPv6アドレスを指定 |
| **CNAME** | ドメインのエイリアス（別名）を設定 |
| **MX** | メールサーバーを指定 |
| **NS** | ドメインのネームサーバーを指定 |
| **TXT** | テキスト情報を格納（SPF、DKIM等で使用） |
| **SOA** | ゾーンの管理情報を格納 |
| **PTR** | 逆引き（IPアドレスからドメイン名）に使用 |
| **SRV** | 特定サービスのポート情報を指定 |

## DNSのゾーン

**ゾーン**とは、DNSの管理単位のことです：

- 特定のドメインとそのサブドメインをまとめて管理
- 権威サーバーがゾーンファイルとして情報を保持
- ゾーン転送により、プライマリ/セカンダリサーバー間で同期

## クラウドにおけるマネージドDNSサービス

各クラウドプロバイダーは、マネージドDNSサービスを提供しており、これらは**権威ネームサーバー**として機能します。

### ✅ AWS Route 53

**Amazon Route 53** は、AWSが提供するスケーラブルなDNSサービスです。

#### 主な機能
- **パブリックホストゾーン**: インターネット公開用のDNS
- **プライベートホストゾーン**: VPC内部のみで使用するDNS
- ヘルスチェックとフェイルオーバー機能
- トラフィックポリシーによる高度なルーティング

#### 使用例
```
# パブリックホストゾーン
example.com
  ├─ www.example.com → ELBのパブリックIP (A レコード)
  └─ api.example.com → EC2のElastic IP (A レコード)

# プライベートホストゾーン（VPC内のみ）
internal.local
  ├─ db.internal.local → RDSのプライベートIP (A レコード)
  └─ cache.internal.local → ElastiCacheのエンドポイント (CNAME レコード)
```

### ✅ GCP Cloud DNS

**Cloud DNS** は、GCPが提供する高パフォーマンスなDNSサービスです。

#### 主な機能
- **パブリックゾーン**: グローバルに公開するDNS
- **プライベートゾーン**: VPC内部のみで使用するDNS
- Cloud Load Balancingとのネイティブインテグレーション
- DNSSEC対応

#### 使用例
```
# パブリックゾーン
example.com
  ├─ www.example.com → Cloud Load Balancerの外部IP
  └─ storage.example.com → Cloud Storageバケット (CNAME)

# プライベートゾーン
internal.example.com
  ├─ db.internal.example.com → Cloud SQLのプライベートIP
  └─ app.internal.example.com → GKEサービスの内部IP
```

## DNSサーバの配置とドメイン登録の仕組み

### DNSサーバはネットワークのどこにいるか？

各DNSサーバは、以下のように配置されています：

```
インターネット全体
├─ ルートサーバ (13系統)
│   └─ 世界中に数百台が分散配置（Anycast技術で最寄りサーバに自動接続）
│
├─ TLDサーバ (.com, .jp, .org など)
│   └─ レジストリ運営組織が管理（例: Verisign, JPRS）
│   └─ 各TLDごとに複数台が世界中に分散
│
├─ 権威ネームサーバ (あなたのドメイン用)
│   └─ ドメイン所有者が管理または委託
│   └─ クラウドプロバイダー（Route 53, Cloud DNS）や
│   └─ DNSホスティング事業者（Cloudflare, お名前.comなど）のデータセンター
│
└─ リカーシブリゾルバ
    └─ ISPや企業ネットワークが運営
    └─ Google Public DNS (8.8.8.8), Cloudflare DNS (1.1.1.1) など
```

### クラウドDNSサービスへの登録は何に該当するか？

クラウドDNS（Route 53やCloud DNS）にドメインを登録する行為は、**権威ネームサーバへの登録**に該当します。

#### 登録プロセスの全体像

```
1. ドメインレジストラでドメインを購入
   ↓
   【TLDサーバへの登録】← ここでNSレコードが登録される
   お名前.com、GoDaddyなどのレジストラが
   .com TLDサーバに「example.comの権威サーバはns-xxx.awsdns-xx.comですよ」
   と登録（NSレコード）
   ↓
2. クラウドDNSでホストゾーンを作成
   ↓
   【権威サーバへの登録】← ここで実際のAレコード等を設定
   Route 53やCloud DNSに
   「www.example.com → 203.0.113.10」などのレコードを登録
   ↓
3. 名前解決の流れ
   TLDサーバ「example.comの権威サーバはこちら！」
   → 権威サーバ（Route 53）「www.example.comのIPは203.0.113.10です！」
```

### VPC内のパブリックIPへのドメイン割り当て

VPC内のEC2インスタンスなどのパブリックIPにドメインを割り当てる場合：

```
【パブリックDNSの場合】
Route 53パブリックホストゾーン
  ├─ web.example.com → EC2のElastic IP (52.xxx.xxx.xxx)
  └─ api.example.com → ALBのパブリックIP
  
→ インターネット全体からアクセス可能
→ TLDサーバに登録された権威サーバとして機能

【プライベートDNSの場合】
Route 53プライベートホストゾーン
  ├─ app.internal → EC2のプライベートIP (10.0.1.50)
  └─ db.internal → RDSのプライベートIP (10.0.2.100)
  
→ VPC内からのみアクセス可能
→ TLDサーバへの登録は不要（VPC内で完結）
```

### 重要なポイント

1. **TLDサーバへの登録**: ドメインレジストラ経由で、「どの権威サーバに問い合わせればいいか」（NSレコード）を登録
2. **権威サーバへの登録**: Route 53やCloud DNSで、「具体的にどのIPアドレスに紐付くか」（AレコードやCNAMEなど）を登録
3. **DNSサーバの配置**: 権威サーバ（クラウドDNS）は、クラウドプロバイダーのデータセンター内に配置されており、グローバルに分散している

## セキュリティとベストプラクティス

### DNSSEC ✅
- DNS情報の改ざんを防ぐ電子署名技術
- データの完全性と認証を保証

### 推奨される設定
- 複数のネームサーバーを設定（冗長化）
- 適切なTTL値の設定
- DNSキャッシュポイズニング対策
