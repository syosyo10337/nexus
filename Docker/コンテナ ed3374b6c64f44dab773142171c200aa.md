# 概略

---

Docker Image がスナップショットだとしたらDocker Container は、

その ==**「スナップショットから起動したプロセス」**==です。

具体的には`docker run`を実行する時、

Docker Image をもとにしてDocker Containerが作成され、隔離された環境が作成されます。

Docker Container は Docker Imageを元にして作成され、リソースの許す限り立ち上げることができます。

💡

Docker Container は1つのコマンドをフォアグラウンドで動かすように設計されている

### Docker Container

---

Dockerイメージをもとに作成され、具現化されたファイルシステムとアプリケーションが実行されている状態。

一度作成されたコンテナは同じイメージから作成されたものであっても、それぞれに状態を持ちます。この点において、状態を持たないイメージとは異なります。

生成されたDockerImageはRead Onlyです。 Docker Containerを作成することで変更可能なレイヤーが新しく作成され、その上でプロセスを動かします。

# ポートフォワーディング

---

仮にGoでwebサーバーを起動するコンテナをrunした時、アプリ上のコードではport8080を公開して、アクセスできるようにしているはずだが、ローカル環境から8080にアクセスできない.

```Bash
e.g.)
$ curl http://localhost:8080
curl: (7) Failed to connect to localhost port 8080 after 8 ms: Connection refused
```

Dockerは仮想環境ではあるが、外から独立したひとつのマシンのように扱えます。

つまり、この場合には8080ポートはコンテナポートと呼ばれ、コンテナ内に限定して開放されているポートである。（仮にcurlをこのコンテナ内で実行すれば届く。）

ここで、Dockerの持つ機能のひとつである**ポートフォワードです。**この機能はホストマシンのポートをコンテナポートと紐付けることで、コンテナの外から来た通信をコンテナポートに転送することができます。

```Bash
# 基本書式
$ docker container run -p <ホスト側ポート>:<コンテナポート>
# -p オプジョンに

e.g.)
$ docker container run -d -p 9000:8080 example/echo:latest
# ホスト側の9000で受け取ったのと、コンテナポート8080に転送している
```

また、ホスト側のポートは省略可能で、省略した場合には、ホスト側のエフェメラルポートが自動割り当てされる

```Bash
e.g.)
$ docker container run -p 8000 example/echo:latest
#$ docker container lsのPORTSで割り当てポート番号を確認できる
```

コンテナポートに対するイメージとしては、ホストとコンテナの間に一枚壁があり、コンテナはコンテナホスト上に起動しているイメージ。

つまりdindを仮に行った時には、仮想dockerホスト上のコンテナにアクセスするまでに、**ホスト　→ 仮想docker host(ホストマシンから見るとコンテナ) → 仮想dockerホスト上のコンテナ**

と2つのコンテナを経由することになるので、2回port fowardingが必要になります。

## ライフサイクル

---

Docker コンテナは実行中、停止、廃棄という3つの状態のいずれかに分類されます。

より厳密には5つの状態を持つとも言える？

### 3分類

- 実行中　—以下の実行中と同じ。CMD ENTRYPOINTで定義されたアプリケーションを実行している間がこの状態です。

実行完了すると、停止に移行します。

- 停止 —実行中のコンテナを明示的に停止(docker container stop)したり、アプリが異常終了した場合はこの状態になります。**ディスクにコンテナ終了時の状態を保持されています。再度実行可能です。**

- 破棄 —停止されたコンテナは明示的に吐きしない限りディスクに残り続けます。逆に一度削除してしまうと、全く同じ状態をもつコンテナを再度生成することはできません。

[![](Screen_Shot_2022-10-19_at_16.58.06.png)](%E3%82%B3%E3%83%B3%E3%83%86%E3%83%8A/Screen_Shot_2022-10-19_at_16.58.06.png)

1. `Image`
    
    - 指定したDocker Image からDocker Containerを起動します。
    

2. `RUNNING`(up)
    
    - Docker Containerが起動した状態です。
    
    - Dockerfileの `CMD` もしくは `ENTRYPOINT` で指定したコマンドがフォアグラウンドで動いている間がRUNNINGの状態です。
    
    - 例えば `docker run -P nginx` のようにnginxを起動した場合、nginxが起動してアクセスを待ち受けてる間はRUNNINGの状態となります
    

3. `STOPPED` (exited)
    
    - 起動したContainerが終了した状態です。
    
    - 正常終了・異常終了、どのような形であっても終了したContainerはSTOPPEDへ遷移します。この時ディスクにコ
    

4. `PAUSED`
    
    - Containerが停止した状態です。
    
    - ユーザーが `docker pause <CONTAINER ID>` を実行すると、現在の状態を保持して一時停止します。
    
    - `docker unpause <CONTAINER ID>` で一時停止したコンテナIDを指定することで再開することが可能です。
    
    - ユーザーが明示的に指定しない限りこの状態へは遷移しません。
    

5. `DELETED`
    
    - Docker Container は明示的に削除を行わない限り停止した状態で残り続けます。
    
    - `docker rm <CONTAINER ID>` で明示的に削除するとDELETEDの状態へ遷移し、削除されます。

## Unison FileSystem(Dockerのファイルシステム)

---

[![](Screen_Shot_2022-11-12_at_16.24.56.png)](%E3%82%B3%E3%83%B3%E3%83%86%E3%83%8A/Screen_Shot_2022-11-12_at_16.24.56.png)

Docker Container から Docker Image へファイルを読み込む際、気をつけないとオーバーヘッドが大きくなります。Containerレイヤーに操作対象のパスが存在しない場合、Imageレイヤーにファイルが無いかの捜査を行います。この捜査は1レイヤーごとに見ていくため、レイヤーが多くなればなるほどオーバーヘッドが大きくなっていきます。

また、Dockerがデフォルトで使用しているファイルシステムではCopy On Write方式でファイルの読み書きを行います。ファイルの更新がかかる度に捜査を実行するため、

==ログのように書き込みの激しいパスはDataVolumeを使用してUnison FileSystemを回避すると良いでしょう。==

イメージは1つのイメージで完結しているわけではなく、複数のイメージを積み重ねて最終的に1つのイメージになります。

また、積み重ねられたイメージのことを中間イメージと呼びます。 どのような歴史でイメージがビルドされたかを知るには

```Bash
$ docker history
```

を使用することで可能です。

### **レイヤーと中間イメージ**

レイヤーと中間イメージは別物です。Dockerfile上でコマンド( `EXPOSE` , `ENV` , `COPY` , etc)を実行するたびに中間イメージが作成されます。レイヤーは `RUN` , `COPY` , `ADD` の実行時に増やし、UnisonFileSystemへの影響があります。基本的に、中間イメージではなくレイヤーを増やさないこと意識するのが良いでしょう。
