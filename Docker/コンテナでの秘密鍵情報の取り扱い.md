# コンテナでの秘密鍵情報の取り扱い

[

Dockerイメージビルド時の秘密情報の扱い方に関するまとめ

本記事ではDockerにおける秘密情報を (I) コンテナを起動する際に使用する秘密情報 と(II) イメージをビルドする際に必要となる秘密情報 に分類して考え、特に後者を安全に取り扱うための方法について整理します。 (I) コンテナを起動する際に使用する秘密情報としては、例えば の環境変数 MYSQL_ROOT_PASSWORD があります。コンテナの起動時にこの環境変数を与えると MySQL の root ユーザーのパスワードがその値になるというものです。 あるいは、例えば Laravel などのWebアプリケーションで、バックエンドDBへの接続情報を含んだ設定ファイルを Docker ホスト側に用意しておき、コンテナ起動時にその設定ファイルをマウントするというようなケースも考えられます。 いずれにしても、このような秘密情報は Docker イメージの中には含めず、コンテナの起動時に環境変数やファイルマウントの形で Docker ホストから渡すようにするのが慣例となっています。 また、このようなタイプの秘密情報を管理する仕組みとして Docker Compose の secrets や Kubernetes の Secret などがあります。 本記事ではこれ以上の詳細に立ち入りませんが、 これらの仕組みは (II) のタイプの秘密情報を扱うためのものではありません 。 (II) イメージをビルドする際に必要となる秘密情報 としては、例えば GitHub のプライベートリポジトリからソースコードを pull する際、認証のために使用するSSH秘密鍵 Amazon S3 のプライベートなバケットに保存されたファイルをダウンロードするために使用するアクセスキー

# 問題

Capistranoでデプロイしたいのだが、

コンテナの中に完全な開発環境を持つことにするべきかということ。

つまり、app/ディレクトリの外にある。.sshの秘密鍵情報などをコンテナにマウントさせて管理すべきなのか？ということ。

そうしなければ、以前のようにdockerの外から、git pushなどを行うことになるだろう。

## 考えられること

1.) secret等のdockerのコマンドや、COPYなどを駆使して、コンテナに機能を持たせてしまう。

### 問題、

機密な情報がコンテナの中に入ってしまう。

→マウントさせずに

```Ruby
docker cp
```

などを単に行うだけだと、app/ディレクトリのスコープの外は毎度毎度入れ直さないと、反映されない。

かと言いて、COPYなどでコンテナ内に永続化させたくはない気がする。

　**しかし、**railsのアプリコードでの設定を行う場合には、コンテナが完全な開発環境である必要がある。(コードの実行環境(dockerコンテナ内)で参照できないローカルのファイルがあるといけないという意味。)

　コンテナ内とホスト側で対応関係も自ずから必要(gemがインストールされている必要があるなど)なので、ローカル内に、二つ開発環境を持っていることになり得る。何だか、無駄な気がする。

## 実際にやったこと。

1. dockerにvolumesコンテナを作る。

2. その中に、 `docker containr cp`を使ってコンテナにsshキーを収める。

```Ruby
#docker-compose.yml

rails:
	volumes:
		config:/root/.ssh/

volumes:
	config
```

←実際にこのコンテナ環境を使った上でデプロイする際には、key-genをコンテナ上でやることになるため。問題ないと考える。
