# Dockerfile

Docker イメージは読み取り専用のレイヤーにより構成されます。 個々のレイヤーは Dockerfile の各命令を表現しています。

```Docker
# syntax=docker/dockerfile:1
FROM ubuntu:18.04
COPY . /app
RUN make /app
CMD python /app/app.py。
```

各コマンドからは 1 つずつレイヤーが生成されます。  
FROMは Docker イメージubuntu:18.04からレイヤーを 1 つ生成します。  
COPYは Docker クライアントのカレントディレクトリからファイルをコピーします。  
RUNはmakeを使ってアプリケーションをビルドします。  
CMDはコンテナー内にて実行するコマンドを指定します

## 基本的なコマンド

---

e.g.）Dockerfile

```Docker
FROM node:19-apline

ENV NODE_ENV=production

WORKDIR /scripts

COPY . .

RUN npm install \
	&& groupadd app \
	&& useradd -g app -m app \
	&& mv /root/.config /home/app/ \
	&& chown -R app:app /scripts /home/app/.config
USER app

CMD ["npm", "start"]
# CLIだと $ npm start と同義
```

### `FROM`

ベースとなるDocker Image を指定します。DockerはベースとなるDocker Image の上に `COPY` や `RUN` のようなコマンドを重ねてDocker Image を作成します。ベースとなるDocker Image は公式で提供されているImageを使用するのが一般的です。ここで指定されるイメージを==ベースイメージ==という。`:19-apline`のように`:`以降に記述される部分はタグと呼び、各イメージのバージョンなどに関する識別子です。==git commitにタグつけるみたいに識別子あったほうがいいよ==

✅

FROM宣言の後には、引数がリセットされてしまうので、変数を使い時は、再度宣言する必要があります。

### `ENV`

Dockerコンテナ内で使用する環境変数を定義します。`NODE_ENV`のようなDockerの起動時にデフォルトで定義されていてほしい環境変数を定義すると良いでしょう。(ubuntuでいうUSERとかとか)

- `WORKDIR`

Dockerfileでコマンドを実行する際に基準となるディレクトリを設定します。

このとき存在しないディレクトリを指定すると自動的にディレクトリが作成されます。(上の例だと、/scriptsディレクトリにて、コマンドを実行することになる　）

デフォルトだと `/`が設定されているため、最悪の場合既存のディレクトリを上書きしてしまいコンテナが起動しなくなります

### `COPY`

Dockerを動作させているホストマシンにあるファイルやディレクトリを、コンテナ内のファイル/ディレクトリにコピーする指定です。

```Docker
 COPY <ホスト側のディレクトリ>　<ドッカー側のディレクトリ> 
```

cf) `ADD`

ホスト側のディレクトリは `docker build .`で指定したディレクトリです。この場合 `.`を指定しており、カレントディレクトリが参照されます。

Docker側は `WORKDIR` で定義されたディレクトリを参照します。

## `RUN`

Dockerfileをもとに、実際にDockerイメージをビルド時する際に、Dockerコンテナ内で実行するコマンドを定義する。（つまり当該のイメージがコンテナとして実行される時のコマンドを記述する）

引数には、コンテナ内で実行したいコマンドをそのまま指定します。==e.g.) mkdir /echo==

ここでコンテナへ依存するライブラリやパッケージのインストールやユーザーの設定などの処理を実行することが多いでしょうね。

### `--mount=type=bind (bind mount)`

ビルドコンテキストからファイルやディレクトリを一時的にマウントする機能

### `**--mount=type=cache**` (Cache Mount)

パッケージマネージャーのキャッシュディレクトリをビルド間で永続化する機能 [Dockerbuild](https://dockerbuild.com/tutorials/docker-caching)[Dockerbuild](https://dockerbuild.com/blog/docker-caching-techniques)

*npmの場合、`/root/.npm`がデフォルトのキャッシュディレクトリです。

- キャッシュマウントはビルド時のみ利用可能で、最終イメージには含まれません [GitHub](https://github.com/moby/buildkit/issues/2147)

- Docker レイヤーキャッシュが無効化されても、キャッシュマウントのディレクトリ内容はビルド間で保持されます [Depot](https://depot.dev/blog/how-to-use-buildkit-cache-mounts-in-ci)

- キャッシュはBuilderごとに管理され、異なるBuilderでは共有されません

### `USER`

作成したDocker Image を起動時にログインするユーザーを指定します。

デフォルトは`root`が設定されているため、セキュリティリスクを回避するために別のユーザーを指定するのが良いでしょう。

[https://www.docker.com/ja-jp/blog/understanding-the-docker-user-instruction/](https://www.docker.com/ja-jp/blog/understanding-the-docker-user-instruction/)

### `CMD`

Dockerコンテナとして実行時にコンテナ内で実行するプロセスを指定します。イメージをビルドするための`RUN`に対して、`CMD`はコンテナ起動時に一度実行するもの。こちらでアプリケーションを動作させるイメージ。

CMDは実行時に上書きできる。　

```Bash
$ docker container run $(docker image build -q .) echo yay
# CMDをecho yayにコマンドを上書きしている。
```

==Dockerはここで設定したコマンドがフォアグラウンドで実行されている間が生存期間になります。==

そのため、プロセスの処理が走っている間はフォアグラウンドで実行するように記述します(バックグラウンドで起動するとDockerが終了してしまう)。

✅

**CMDの3つの指定方式  
1. CMD [”実行ファイル”, “引数1”, “引数2”] (推奨)**  
**2. CMD コマンド　引数1 引数2**  
—シェル内で実行される。そのため、シェル変数などを引き継いでいる  
**3. CMD 引数1 引数2**  
—ENTRYPOINTの引数を指定する。

## よく使用されるコマンド

- `****EXPOSE****`

コンテナ起動時に公開することを想定されているポートを記述します。

記載することで他の人から「このDockerはどのポートを使用するのか」がわかりやすくなるため、記述すると丁寧でしょう。

```Docker
EXPOSE 3000
```

???

コンテナ起動時に `EXPOSE` で指定されたポートをホスト側へ公開するには `-P` オプションを使用する必要があります。

```Bash
$ docker run -P nginx
```

### `**VOLUME**`

Data Volumeを作成するためのコマンドです。Volumeについては後の章で説明します。永続的なデータや共有するためのデータ、更新頻度の激しいファイルを扱うために使用されます。基本的に永続的なデータはDockerで管理することは推奨されないため、基本的にログのような更新頻度の激しいファイルで使用すると良いでしょう。

```Docker
# コンテナ内のvolumeの位置を指定？
VOLUME ["/app/log"]
```

### `ARG`

Dockerfileからイメージをビルド時(docker buildコマンドを打つ時)に、使用できる変数を指定できる。シェルに渡す環境変数ではなく、build時の一時的な変数としての役割を持たせられる。

==ビルドの前提条件/必要情報が増えると複雑化につながるため、基本的に使用しない方が良いです。==

```Docker
ARGS ${node_env:-production}
ENV node_env
```

```Bash
$ docker build --build-arg node_env=development .
```

- `ADD`

`COPY` コマンドを拡張したコマンドです。主に以下の3つの機能を持ちます。

1. `COPY` と同じく指定したパスをコンテナ内へコピー

2. URLを指定した場合、そのURLからファイルをダウンロードし、コンテナ内へコピー

3. コピーされたパスが `.tar` もしくは `.tar.gz` の場合解凍する

💡

基本的に`ADD`を使うのは避け、`COPY`を使うのが良いでしょう。

`ADD`は

「指定したファイルのコピー」・

「指定したURLの取得 」・

「圧縮されたファイルの解凍」と非常に高機能な反面、Dockerfileの複雑度が増します。

また、URLからファイルを取得するのはパブリックへ依存が発生し、URLの参照先が変更された際にビルド不可なDockerfileができてしまいます。`COPY`の場合機能がシンプルなのでシンプルなDockerfileの作成に繋がります。

### `ENTRYPOINT`

コンテナを起動(run)した時に必ず実行されるコマンドを指定するための記法。

shellフォームとexecフォームの書き方が存在する。

`docker run`時に指定したコマンドを `ENTRYPOINT`の引数として使用します。

```Docker
ENTRYPOINT ["echo"]
```

引数に "hello!" を渡すと `echo "hello!"`が実行される

```Bash
e.g.1)

$ docker run <IMAGE NAME> "hello!"
>hello!
```

この `ENTRYPOINT` に記載されたものはコンテナrun時に基本的に書き換えられることはなく、オプションとして追加されるような形で追記される。 `CMD`で設定されたものについてはコンテナrun時に書き換えられる。

```Bash
e.g.2)
#(CMDの上書きのように)ENTRYPOINTも上書きする例
$ docker run --entrypoint date <IMAGE NAME>
Mon Mar 18 22:36:19 JST 2019

# --entrypointオプションで上書きしている。dateコマンドが実行された
```

💡

**ENTRYPOINT vs CMD**

基本的に `CMD` を使うのが良いでしょう。`ENTRYPOINT` はDocker起動時のコマンドを強制します。コマンドのラップをするDocker Image の場合は `ENTRYPOINT` のほうが好ましいですが、一般的なWebアプリケーションの場合は `CMD` を使用する方がユーザーにとって使いやすいDocker Image になります。

# 実装例

### Next.js版で書いてみる

- 単一Dockerfile版
    
    ```Docker
    # syntax=docker/dockerfile:1
    # NOTE: birdcageでは、単一Dockerfileの構成を採用し,環境ごとに利用するstageを切り替えている。
    
    # TODO: sbがnode24対応完了したら、node:24.xに更新する cf. https://github.com/storybookjs/storybook/issues/31434
    FROM node:23.11.0-slim AS base
    WORKDIR /opt/birdcage
    
    
    # =========== depsステージ ===========
    FROM base AS deps
    
    # HACK: npmバージョン固定(baseイメージのdefaultが10.9.2であるため。)
    # NOTE: deps/devステージでnpm@11.4.2を使用。builderはbaseのnpm（10.9.2）を使用するが、
    #       node_modulesは11.4.2でインストール済みなので問題ない
    RUN npm install -g npm@11.4.2
    
    # 後続処理で参照する可能性があるのと、サイズが小さいため、--mount=type=bindを使用しない
    COPY package.json package-lock.json ./
    RUN --mount=type=cache,target=/root/.npm,sharing=locked \
        npm ci
    
    
    # =========== devステージ ===========
    FROM deps AS dev
    
    # dockerコンテナ内で、codemodなどを利用するときにリポジトリの所有者とプロセス実行者が違ってエラーになるため
    # NOTE: git configコマンドを実行するためだけにgitをインストール
    # 実際にgitが必要かどうかは、使用状況に応じて削除を検討
    RUN apt-get update && \
        apt-get install -y --no-install-recommends git && \
        git config --global --add safe.directory /opt/birdcage && \
        rm -rf /var/lib/apt/lists/*
    
    ENV NODE_ENV=development
    EXPOSE 3000 6006
    CMD ["npm", "run", "dev"]
    
    
    # =========== builderステージ ===========
    FROM base AS builder
    
    COPY --from=deps /opt/birdcage/node_modules ./node_modules
    COPY . .
    RUN --mount=type=cache,target=/opt/birdcage/.next/cache,sharing=locked \
        npm run build
    # NOTE: 開発環境以外ではmswは利用しないので該当ファイルを削除する
    RUN rm public/mockServiceWorker.js
    
    # =========== runnerステージ ===========
    FROM base AS runner
    USER 1000
    ENV NODE_ENV=production
    
    # standaloneビルド出力をコピー
    # cf.https://nextjs.org/docs/pages/api-reference/config/next-config-js/output#automatically-copying-traced-files
    COPY --chown=1000:1000 --from=builder /opt/birdcage/.next/standalone ./
    COPY --chown=1000:1000 --from=builder /opt/birdcage/.next/static ./.next/static
    COPY --chown=1000:1000 --from=builder /opt/birdcage/public ./public
    
    EXPOSE 3000/tcp
    ENV PORT=3000
    ENV HOSTNAME="0.0.0.0"
    CMD ["node", "server.js"]
    ```
    

- ローカル開発の例
    
    watchを使って、devContainer開発（host→containerの同期が不要な場合）volume trickを使わない。
    
    ```Docker
    # syntax=docker/dockerfile:1
    # TODO: sbがnode24対応完了したら、更新する cf. https://github.com/storybookjs/storybook/issues/31434
    FROM node:23.11.0-slim AS base
    
    # 依存関係インストールステージ
    FROM base AS deps
    
    WORKDIR /app
    
    # 必要なツールをインストール（slimはgitが含まれていない）
    RUN apt-get update && \
        apt-get install -y --no-install-recommends git && \
        rm -rf /var/lib/apt/lists/*
    # npmバージョンを固定
    RUN npm install -g npm@11.4.2
    
    # 依存関係のみインストール
    # マルチステージ市にしない場合は、起動前にpackageが入っている必要があります。
    RUN --mount=type=bind,source=package.json,target=package.json \
        --mount=type=bind,source=package-lock.json,target=package-lock.json \
        --mount=type=cache,target=/root/.npm,sharing=locked \
        npm ci
    
    
    # 開発ステージ
    FROM deps AS dev
    # dockerコンテナ内で、codemodなどを利用するときにリポジトリの所有者とプロセス実行者が違ってエラーになるため
    RUN git config --global --add safe.directory /app
    
    COPY . .
    
    EXPOSE 3000 6006
    ENV NODE_ENV=development
    CMD ["npm", "run", "dev"]
    
    ```

# テストコマンド

以下のようなファイルを配置してimageサイズを計測すると良さそう。

```YAML
#!/bin/bash
# build-metrics.sh
# Usage: ./bin/build-metrics.sh [environment] [cache]
#   environment: feature|local|production|unified (optional, default: all)
#   cache: true|false (optional, default: true)
#
# Examples:
#   ./bin/build-metrics.sh                    # すべての環境でキャッシュありビルド
#   ./bin/build-metrics.sh feature            # featureのみキャッシュありビルド
#   ./bin/build-metrics.sh local false        # localのみキャッシュなしビルド
#   ./bin/build-metrics.sh production true    # productionのみキャッシュありビルド
#   ./bin/build-metrics.sh unified            # unifiedのdev/runnerステージをビルド

set -e

# 引数の解析
TARGET_ENV="${1:-all}"
USE_CACHE="${2:-true}"

# 利用可能な環境
AVAILABLE_ENVS=("feature" "local" "production" "unified")

# 環境のバリデーション
if [[ "$TARGET_ENV" != "all" ]] && [[ ! " ${AVAILABLE_ENVS[@]} " =~ " ${TARGET_ENV} " ]]; then
    echo "❌ Error: Invalid environment '${TARGET_ENV}'"
    echo "Available environments: feature, local, production, unified, all"
    exit 1
fi

# キャッシュオプションのバリデーション
if [[ "$USE_CACHE" != "true" ]] && [[ "$USE_CACHE" != "false" ]]; then
    echo "❌ Error: Invalid cache option '${USE_CACHE}'"
    echo "Cache option must be: true or false"
    exit 1
fi

# ビルド対象の環境を決定
if [[ "$TARGET_ENV" == "all" ]]; then
    BUILD_ENVS=("${AVAILABLE_ENVS[@]}")
else
    BUILD_ENVS=("$TARGET_ENV")
fi

# キャッシュオプションの設定
CACHE_FLAG=""
CACHE_LABEL="(with cache)"
if [[ "$USE_CACHE" == "false" ]]; then
    CACHE_FLAG="--no-cache"
    CACHE_LABEL="(no cache)"
fi

echo "=== Docker Build Metrics ==="
echo "Target: ${TARGET_ENV} | Cache: ${USE_CACHE}"
echo ""

# unified Dockerfileのステージ別ビルドとメトリクス収集
build_unified_stage() {
    local stage=$1
    local dockerfile="dockerfiles/Dockerfile"
    local tag="birdcage:unified-${stage}"
    local stage_label=""

    case "$stage" in
        "dev")
            stage_label="dev (local開発用)"
            ;;
        "runner")
            stage_label="runner (feature/production用)"
            ;;
        *)
            echo "  ❌ Unknown stage: ${stage}"
            return 1
            ;;
    esac

    echo "📦 Unified Dockerfile - ${stage_label} ${CACHE_LABEL}"
    echo "Building..."

    # ビルド時間の計測
    START=$(date +%s)
    if docker build -f "$dockerfile" --target "$stage" -t "$tag" $CACHE_FLAG --progress=plain . > /dev/null 2>&1; then
        END=$(date +%s)
        BUILD_TIME=$((END - START))

        # イメージサイズの取得
        SIZE_BYTES=$(docker image inspect "$tag" --format='{{.Size}}')
        SIZE=$(echo "$SIZE_BYTES" | numfmt --to=iec-i)

        # レイヤー数の取得
        LAYERS=$(docker image inspect "$tag" --format='{{range .RootFS.Layers}}{{println .}}{{end}}' | wc -l)

        # 結果の表示
        echo "  ✓ Build Time: ${BUILD_TIME}s"
        echo "  ✓ Image Size: ${SIZE} ($(numfmt --to=si $SIZE_BYTES))"
        echo "  ✓ Layers: ${LAYERS}"
    else
        echo "  ❌ Build failed"
        return 1
    fi

    echo ""
}

# 各環境のビルドとメトリクス収集
build_and_measure() {
    local env=$1
    local dockerfile="dockerfiles/Dockerfile"
    local tag="birdcage:${env}"

    echo "📦 ${env^} Dockerfile ${CACHE_LABEL}"
    echo "Building..."

    # ビルド時間の計測
    START=$(date +%s)
    if docker build -f "$dockerfile" -t "$tag" $CACHE_FLAG --progress=plain . > /dev/null 2>&1; then
        END=$(date +%s)
        BUILD_TIME=$((END - START))

        # イメージサイズの取得
        SIZE_BYTES=$(docker image inspect "$tag" --format='{{.Size}}')
        SIZE=$(echo "$SIZE_BYTES" | numfmt --to=iec-i)

        # レイヤー数の取得
        LAYERS=$(docker image inspect "$tag" --format='{{range .RootFS.Layers}}{{println .}}{{end}}' | wc -l)

        # 結果の表示
        echo "  ✓ Build Time: ${BUILD_TIME}s"
        echo "  ✓ Image Size: ${SIZE} ($(numfmt --to=si $SIZE_BYTES))"
        echo "  ✓ Layers: ${LAYERS}"
    else
        echo "  ❌ Build failed"
        return 1
    fi

    echo ""
}

# 各環境のビルドを実行
for env in "${BUILD_ENVS[@]}"; do
    if [[ "$env" == "unified" ]]; then
        # unifiedの場合はdevとrunnerステージをビルド
        build_unified_stage "dev"
        build_unified_stage "runner"
    else
        build_and_measure "$env"
    fi
done

echo "=== Build Metrics Completed ==="
```
